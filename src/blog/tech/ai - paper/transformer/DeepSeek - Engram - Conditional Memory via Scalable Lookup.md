*论文发布时间：2026-01-12*

DeepSeek 发布的 **Engram**（论文标题为《[Conditional Memory via Scalable Lookup: A New Axis of Sparsity for Large Language Models](https://arxiv.org/abs/2601.07372)》）在 AI 圈引起了很大关注。
简单来说，这篇论文提出了一种**"条件存储"（Conditional Memory）**机制，旨在为大语言模型（LLM）增加一个像"词典"或"百科全书"一样的外部查询模块，让模型不再需要把所有知识都死记硬背在神经元的权重里。

## Engram 解决了什么问题？

**核心问题：Transformer 模型在处理"静态知识"时极度低效。**

目前主流的 Transformer（包括 GPT-4 和 DeepSeek-V3 这种 MoE 模型）存在一个根本缺陷：**它们缺乏原生的"查询"能力。**

* **计算浪费：** 当模型遇到一个常见的短语或事实（比如"巴黎的首都是..."）时，它依然需要动用昂贵的注意力机制（Attention）和全连接层（FFN）去进行复杂的矩阵运算，从权重中"重新计算"出答案。
* **存储压力：** 为了记住海量事实，模型必须不断增加参数量。这导致显存（HBM）需求激增，计算成本极高。

**Engram 的作用：** 它引入了一个  复杂度的查找表。模型在处理文本时，可以通过哈希（Hashing）直接从一个海量的 N-gram 嵌入表中"提取"相关知识，而不需要经过层层神经网络的重度计算。

## 这个问题真实存在吗？

**真实存在且是目前大模型进一步扩展（Scaling）的头号瓶颈。**

* **机械记忆 vs. 逻辑推理：** 研究发现，模型的前几层往往在做"低级"的模式识别和事实提取，后几层才在做真正的"思考"。把事实和逻辑混在一起处理，导致神经元的有效深度被浪费了。
* **硬件瓶颈：** GPU 的显存（HBM）贵且稀缺，但内存（DRAM）和硬盘（NVMe）相对便宜。如果能把"知识"从 GPU 显存里剥离出来，放到 CPU 内存里随用随取，AI 的成本将大幅下降。

## 为什么现在才有人去解决？

其实"检索增强"或"外部存储"的概念一直有（比如 RAG、KNN-LM），但 Engram 解决得更底层、更高效，现在爆发主要有三个原因：

1. **MoE 技术的成熟：** 现在的模型已经通过混合专家（MoE）实现了"条件计算"（只激活部分神经元）。DeepSeek 认为除了计算可以"稀疏化"，**存储也应该"稀疏化"**。
2. **PCIe 带宽的进步：** 过去通过 CPU 内存读取数据太慢。但 Engram 采用了**确定性寻址**和**异步预取**技术，可以在模型计算的同时把知识从内存拉到 GPU，几乎零延迟。
3. **大模型 Scaling Law 的瓶颈：** 盲目堆参数已经边际效应递减了。DeepSeek 发现了一种新的 **U 型比例定律（U-shaped scaling law）**：将 20%-25% 的参数分配给 Engram（存储），剩下的给计算，性能远好于纯计算模型。

## 它是如何解决的？

Engram 的架构包含几个核心组件。为了理解它的实现细节，我们需要同时看清 **Transformer 的软件结构**和**服务器的硬件存储层级**，因为 Engram 的核心就是把两者桥接起来。

### 1. 物理位置：插入在哪里？（软件 × 硬件全景）

![Figure 1: Engram 架构总览。模块通过检索静态 N-gram 记忆并借助上下文感知门控将其与动态隐藏状态融合来增强骨干网络。仅应用于特定层以解耦存储与计算。](https://arxiv.org/html/2601.07372v1/x1.png)

> **图中关键路径解读（三个阶段）：**
>
> **阶段 ①　查表：** 图的左半部分——N-gram 经多头哈希查 DRAM 嵌入表，子向量 Concat 拼接为 $e_t$（1280 维）。
>
> **阶段 ②　门控决策：** Concat 之后的两个 Linear 分别是 $\mathbf{W}_K$ 和 $\mathbf{W}_V$，但它们的输出**并不直接交互**——
> - **$\mathbf{W}_K$（Linear → Key）**：将 $e_t$ 投影为 $k_t$（2560 维），作为记忆的"签名"。$k_t$ 唯一的用途是与当前隐藏状态 $h_t$ 做**点积**，经 sigmoid 算出一个**标量**门控值 $\alpha_t \in (0,1)$——这就是"这条记忆整体要不要用"的开关。$k_t$ 算完就"用完即弃"。
> - **$\mathbf{W}_V$（Linear → Value）**：将 $e_t$ 投影为 $v_t$（2560 维），作为记忆的"内容"。$v_t$ 本身不参与门控计算，它只是等着被门控值加权。
>
> 图中两个 Linear 输出之间的 **⊗** 符号，实际上是 $\alpha_t$（标量）$\times$ $v_t$（向量）＝ $\tilde{v}_t$。因为 $\alpha_t$ 是一个**单个数字**，所以 $v_t$ 的每个维度都乘以**同一个系数**——这是对记忆**整体**的"音量旋钮"，不是逐维度的独立开关。
>
> **阶段 ③　卷积精炼：** 门控后的 $\tilde{v}_t$ 经过 Conv1D（因果深度卷积）让相邻位置的 Engram 输出做一点局部交流，再通过残差连接得到最终输出 $Y_t$，注入到 $H^{(\ell)}$ 中。
>
> 这两个 Linear 是 Engram 中**仅有的**两个大型可学习投影矩阵（各 2560×1280），详见后文[门控机制详解（C 节）](#C-门控机制详解一个微型-Attention)。

Engram 模块被插入在 Transformer 的 **FFN** 旁边，作为一个**并行**的分支。但关键是——Engram 的查找表**不住在 GPU 显存（HBM）里**，而是住在便宜得多的 CPU 主内存（DRAM）中。这正是它能"降本"的核心：

```
┌──────────────────── 硬件视角 ─────────────────────────────┐
│                                                            │
│  ┌─── GPU (HBM 显存，~80GB，贵) ──────────────────┐       │
│  │                                                 │       │
│  │  Layer 1:  Attention → FFN                      │       │
│  │  Layer 2:  Attention → FFN + α·Engram(x) ◄──┐  │       │
│  │  Layer 3:  Attention → FFN + α·Engram(x) ◄──┤  │       │
│  │  Layer 4:  Attention → FFN                   │  │       │
│  │  ...                                         │  │       │
│  │  Layer N:  Attention → FFN                   │  │       │
│  │                                              │  │       │
│  │  模型权重、激活值、KV-Cache 都在这里          │  │       │
│  └──────────────────────────────────────────────┘  │       │
│                                             PCIe   │       │
│                                             总线   │       │
│  ┌─── CPU (DRAM 主内存，~1TB，便宜) ───┐    ▲     │       │
│  │                                      │    │     │       │
│  │  ┌────────────────────────────────┐  │    │     │       │
│  │  │ Engram 查找表（N-gram 嵌入表）  │──┼────┘     │       │
│  │  │ 数十亿条向量，占数十 GB         │  │          │       │
│  │  │ 每次只取出极小一部分送到 GPU    │  │          │       │
│  │  └────────────────────────────────┘  │          │       │
│  └──────────────────────────────────────┘          │       │
└────────────────────────────────────────────────────────────┘
```

对比标准 Transformer Block 和 Engram 增强 Block 的数据流：

```
【标准 Transformer Block — 全部在 GPU HBM 内完成】

  输入 x → Attention → (+) → FFN → (+) → 输出
                        ↑           ↑
                        x           x

【Engram 增强 Block（Layer 2 或 15）— Engram 在 Attention 之前执行】

                    ┌── Engram 模块 ──────────────────────┐
                    │ 哈希查 DRAM → 门控 → 卷积 → Y       │
                    └──────────────────┬─────────────────┘
                                      │ (通过 PCIe 从 DRAM 取数据)
                                      ▼
  输入 x ───────────────────────→ (+) 残差 → Attention → (+) → MoE/FFN → (+) → 输出
                                  ↑                      ↑               ↑
                                  x                      x               x
```

DeepSeek 的研究发现，最佳的插入位置在模型的**浅层（如第 2-4 层）**，而非深层：

![Figure 5: 架构消融实验。深蓝色曲线为层敏感性扫描（单层 Engram 在不同层插入的效果），确认 Layer 2 最优，越深越差。右侧标记为组件消融，展示了多分支集成、Tokenizer 压缩和上下文感知门控的重要性。](https://arxiv.org/html/2601.07372v1/x5.png)

* **软件视角：** 事实回忆（如"巴黎在法国"）是一种**"低级反射"**，而非高级推理。让浅层通过 Engram 快速查表解决死记硬背的任务，可以**解放深层网络**去专注于复杂的逻辑推演。
* **硬件视角：** Engram 只在浅层触发 DRAM → GPU 的数据搬运，深层完全不涉及外部存储访问，最大限度减少了 PCIe 带宽压力。

### 2. 核心机制：三步走

输入 token 经过 Engram 的完整数据流如下：

```
输入 token: ["白", "日", "依", "山", "尽"]
                    │
       ┌────────────▼────────────┐
  步骤①│  Tokenizer 压缩          │  "The"/"the"/"THE" → 同一个 ID
       │  合并大小写、变体等      │  减少查找表冗余
       └────────────┬────────────┘
                    │  压缩后的 token ID
       ┌────────────▼────────────┐
  步骤②│  构造后缀 N-gram          │  以当前 token 为结尾
       │  + 多头哈希（Multi-Head） │  对每种阶数（2,3）
       │  Hash₁(x), Hash₂(x)... │  × 每个头，共 16 次查表
       └────────────┬────────────┘
                    │  16 个地址
       ┌────────────▼────────────────────────────┐
  步骤③│  查表 + 上下文感知门控                    │
       │                                         │
       │  DRAM 查找表 ──→ 取出 H 个嵌入向量       │
       │         ↓                                │
       │  Gate(当前语境 x) ──→ 权重 0~1           │
       │         ↓                                │
       │  输出 = Gate(x) · Lookup(x)              │
       │  (语境不符则 Gate≈0，自动忽略)             │
       └──────────────────────────────────────────┘
                    │
                    ▼
          残差注入到 H^(l)，再送入 Attention → MoE/FFN
```

*   **Tokenizer 压缩：** 把意义相近的词合并（比如 "Hello" 和 "hello"），减少查找表的冗余。
*   **构造后缀 N-gram + 多头哈希（Multi-Head Hashing）：** 对每个 token 位置，先构造**所有阶数**的后缀 N-gram（Engram-27B 使用 {2,3} 两种阶数），再对每种阶数分别用 K=8 个独立哈希函数查表——**不预判哪条有用，全部查完后交给门控决定取舍**。所有阶数、所有头的结果拼接成一个完整的记忆向量 $e_t$。多头设计同时也缓解了哈希冲突（如"苹果"作为水果 vs 公司，不同头大概率映射到不同槽位）。
*   **上下文感知门控（Context-Aware Gating）：** 查出来的知识不是盲目相加，而是通过一个可训练的门控网络 $\text{Gate}(x)$ 结合当前语境决定权重。如果查出的内容与语境不符，门控会将其权重置为 0，防止噪音干扰。

### 2.5 Engram 模型设计详解

上面的三步走是概念层面的描述。现在我们钻进去看**实际的网络结构**——每一个张量的形状、每一步计算的具体操作。以下以 Engram-27B（论文主力模型）的配置为例。

#### A. 参数预算重分配：FFN 的参数是怎么"割"给 Engram 的？

首先澄清一个常见误解：Engram **不是**把 FFN 的权重矩阵物理拆开，而是在**总参数预算**层面进行重新分配——减少 MoE 路由专家的数量，把省下来的参数额度给 Engram 嵌入表：

```
总参数 P_total = 26.7B（与纯 MoE-27B 完全相同）
激活参数 P_active = 3.8B（每个 token 实际参与计算的参数，也完全相同）

┌─────────────────────────────────────────────────────┐
│  MoE-27B（纯 MoE 基线）                               │
│                                                       │
│  共享骨干（Attention + 2 个共享专家 + ...）              │
│  + 72 个路由专家（top-6 激活）                          │
│    └─ 所有稀疏参数都给了 MoE                           │
│                                          总计 26.7B   │
└─────────────────────────────────────────────────────┘
                       │  重新分配 ↓
┌─────────────────────────────────────────────────────┐
│  Engram-27B（混合分配，ρ ≈ 74.3%）                     │
│                                                       │
│  共享骨干（Attention + 2 个共享专家 + ...） ← 完全不变   │
│  + 55 个路由专家（top-6 激活）  ← 减少 17 个专家        │
│  + Engram 嵌入表                ← 省下的预算给了 Engram  │
│    └─ 5.7B 参数（2 层 × ~2.85B/层）                   │
│                                          总计 26.7B   │
└─────────────────────────────────────────────────────┘
```

分配比例 ρ 控制稀疏参数的去向：

```
  稀疏参数 P_sparse = P_total - P_active = 22.9B

  MoE 专家分得:  ρ × P_sparse     (ρ=74.3% → ~17B 给路由专家)
  Engram 分得:   (1-ρ) × P_sparse (25.7%   → ~5.7B 给嵌入表)

  论文发现的最优区间: ρ ≈ 75%-80%
```

关键点：**每个 token 的实际计算量（FLOPs）完全不变**。MoE 依然只激活 top-6 个专家，Engram 只做 O(1) 查表。改变的只是"仓库里的存货分配"。

#### B. 多头哈希网络的内部结构

Engram 的核心就是这张巨大的哈希查找表。Engram-27B 的配置：d_mem=1280, K=8 个哈希头, N-gram 阶数={2,3}。

```
输入 token 序列（压缩后）: [..., x₃', x₄', x₅']
                                    │
                        构造后缀 N-gram（以位置 t=5 为例）：
                        ┌───────────────────────────┐
                        │ 2-gram: (x₄', x₅')        │
                        │ 3-gram: (x₃', x₄', x₅')   │
                        └────────────┬──────────────┘
                                     │
          ┌──────────────────────────┴──────────────────────────┐
          │                                                     │
    ═══ 2-gram 分支 ═══                                 ═══ 3-gram 分支 ═══
          │                                                     │
  ┌───┬───┼───┬───┐                                   ┌───┬───┬───┬───┐
  H₁  H₂  ...  H₇  H₈                                H₁  H₂  ...  H₇  H₈
  │   │       │   │                                   │   │       │   │
  φ₂,₁ φ₂,₂  φ₂,₇ φ₂,₈    ← 乘法-异或哈希 →        φ₃,₁ φ₃,₂  φ₃,₇ φ₃,₈
  │   │       │   │           (O(1)确定性)             │   │       │   │
  ▼   ▼       ▼   ▼                                   ▼   ▼       ▼   ▼
┌───┐┌───┐ ┌───┐┌───┐                              ┌───┐┌───┐ ┌───┐┌───┐
│E₂₁││E₂₂│ │E₂₇││E₂₈│    ← DRAM 中的子表 →        │E₃₁││E₃₂│ │E₃₇││E₃₈│
│M×d││M×d│ │M×d││M×d│      (M≈226万 × d维)         │M×d││M×d│ │M×d││M×d│
└─┬─┘└─┬─┘ └─┬─┘└─┬─┘                              └─┬─┘└─┬─┘ └─┬─┘└─┬─┘
  │    │      │    │                                   │    │      │    │
  ▼    ▼      ▼    ▼                                   ▼    ▼      ▼    ▼
 [d维] [d维] [d维] [d维]                               [d维] [d维] [d维] [d维]
  │    │      │    │                                   │    │      │    │
  └────┴──────┴────┴───────────────────────────────────┴────┴──────┴────┘
                                     │
                              Concatenate 拼接
                                     │
                                     ▼
                             e_t ∈ ℝ^1280
                          (16 个子向量拼成 1280 维)
```

每个哈希函数 $\varphi_{n,k}$ 的具体实现：
* **类型：** 乘法-异或（multiplicative-XOR）哈希——不是神经网络，是纯数学函数
* **输入：** N-gram 中各 token 的压缩 ID 序列
* **输出：** 一个整数索引（对质数 M 取模，减少冲突）
* **复杂度：** 严格 O(1)，无需矩阵乘法

##### 哈希多头 ≠ Attention 多头：什么是学的，什么不是？

一个容易产生的误解是把 Engram 的"多头哈希"类比成 Attention 的"多头 Q/K/V"。实际上两者完全不同——**哈希函数是固定的，不参与训练**。每个 $\varphi_{n,k}$ 的随机常数在初始化时选定后就不再改变，训练和推理使用**完全相同**的哈希函数和寻址机制。训练和推理的唯一区别是嵌入表的物理位置（训练时在 GPU 上接收梯度更新，推理时迁移到 DRAM 变为只读）。

```
┌─────────────────┬──────────────┬────────────────────────────────┐
│  组件            │  是否可学习？  │  说明                          │
├─────────────────┼──────────────┼────────────────────────────────┤
│  哈希函数 φ      │  ✗ 固定       │  随机常数，初始化后永不改变      │
│  嵌入表向量      │  ✓ 可学习     │  梯度下降优化（Adam，5倍学习率） │
│  W_K, W_V       │  ✓ 可学习     │  门控投影矩阵（后续 C 节详解）  │
│  Conv1D         │  ✓ 可学习     │  零初始化，逐渐学出非零权重      │
└─────────────────┴──────────────┴────────────────────────────────┘
```

两种"多头"的本质区别：

```
Attention 多头（可学习的矩阵乘法）：
  Q = W_Q · x,  K = W_K · x,  V = W_V · x
  → 每个头有独立的可学习权重矩阵
  → 通过矩阵乘法从输入中"提取"不同维度的语义特征

Engram 哈希多头（固定的整数运算）：
  idx = (id₁ × A₁ ⊕ id₂ × A₂) mod M
  → 每个头只是一组不同的随机整数常数 A₁, A₂
  → 通过异或和取模"定位"不同子表中的条目
  → 可学习的部分不在哈希函数本身，而在被查到的那些嵌入向量
```

整个 Engram 模块的数据流因此清晰地分为两个阶段：

```
阶段 1（查表）：N-gram → 固定哈希函数 → 16 个子向量 → Concatenate → e_t ∈ ℝ^1280
                ↑ 哈希函数无可学习参数（但表中的向量是训练学出来的）

阶段 2（门控）：e_t 经 1 套 W_K/W_V 投影 → 与 h_t 点积 → 1 个标量门控 α_t → 卷积 → Y_t
                ↑ W_K, W_V, Conv1D 都是可学习的（详见 C 节）

→ 16 个子向量先拼接成 1 个 e_t，然后只经过 1 套 W_K + 1 套 W_V，
  产生 1 个标量门控值——不存在"每个头单独门控"的操作。
```

表的规模（Engram-27B）：
* 每层 Engram 的总词汇量 ≈ 226 万个 N-gram 条目
* 分布在 16 张子表中（2 阶 × 8 头）
* 两层 Engram（Layer 2 和 Layer 15）各有独立的一整套表
* 总参数：≈ 5.7B，**全部驻留在 DRAM 中**

##### 每个位置都查所有阶数，无意义的 N-gram 怎么办？

一个自然的疑问是：对每个 token 位置都无差别地构造所有阶数的 N-gram，那**跨句、跨段落的拼凑岂不是大概率毫无意义？** 确实如此。比如输入 "今天天气真好。白日依山尽"，处理 token "白" 时构造的 2-gram 是 ("。","白")、3-gram 是 ("好","。","白")——这些跨句组合完全没有语义价值。但 Engram 的设计本身就预期了这种情况，通过三层机制逐级过滤噪声：

```
三层防御机制：

  第 1 层：训练频率淘汰（向量质量）
  ─────────────────────────────────
  训练 262B tokens 后：
    hash("of","the")          → [充分优化的向量]  （出现了数百万次）
    hash("好","。","白")       → [接近随机噪声]    （可能从未出现过）
  → 低频/未见 N-gram 的向量质量很差，本身就很难产生有效信息

  第 2 层：上下文门控（α_t）
  ─────────────────────────────────
  即使查出了向量，门控会计算它与当前语境的匹配度：
    有意义的 ("白","日") + 古诗语境 → α_t ≈ 0.9 → 采纳
    无意义的 ("好","。","白") + 古诗语境 → α_t ≈ 0.05 → 忽略
  → 门控不需要"知道"N-gram 有没有意义，只看查出来的东西合不合语境

  第 3 层：多头稀释
  ─────────────────────────────────
  即使某个头碰巧查到了误导性向量：
    16 个子向量中只有 1 个出问题 → 只影响 80/1280 = 6.25% 的维度
  → 多头拼接天然稀释了单个头的噪声
```

这种"**全查 + 事后过滤**"的哲学和 Attention 完全一致——Attention 也是对所有位置都算相似度，然后 softmax 自动把不相关的位置权重压到接近 0，而不是事先决定"该看哪些位置"。Engram 的门控 $\alpha_t$ 就是同样的逻辑，只不过作用在"查表结果"上。

##### 与 TF-IDF 的设计同构

这套"训练频率决定质量 + 语境决定权重"的机制，和经典信息检索中的 **TF-IDF**（词频-逆文档频率）有着深层的设计同构：

```
┌──────────────┬─────────────────────────────────────────────────┐
│  TF-IDF      │  Engram                                         │
├──────────────┼─────────────────────────────────────────────────┤
│  TF          │  训练频率驱动的向量质量                           │
│  (词频)      │  高频 N-gram 被查数百万次 → 向量充分优化           │
│              │  低频 N-gram 接近随机噪声 → 几乎无效              │
├──────────────┼─────────────────────────────────────────────────┤
│  IDF         │  上下文门控 α_t                                  │
│  (逆文档频率) │  即使向量质量高，若与当前语境不匹配               │
│              │  门控也会将权重压到 ≈ 0                           │
├──────────────┼─────────────────────────────────────────────────┤
│  TF × IDF    │  向量质量 × 门控权重                              │
│  = 最终权重   │  = Engram 的有效输出                             │
│              │  只有"高频且语境相关"的 N-gram 才能真正贡献信息    │
└──────────────┴─────────────────────────────────────────────────┘
```

关键区别在于：TF-IDF 的权重是**人工定义的统计公式**（词频和逆文档频率都有明确的数学定义），而 Engram 的"频率效应"和"语境相关性"都是通过端到端训练**隐式涌现**的——模型自动学会了哪些 N-gram 模式值得存储、在什么语境下该调用它们。可以说，**Engram 是 TF-IDF 精神在神经网络时代的"可微分进化"**：从离散的统计计数升级为连续的梯度优化，从人工定义的公式升级为数据驱动的自适应权重。

#### B+. 知识是怎么存的？又是怎么取的？

上面讲了表的结构，但一个根本问题还没回答：**表里每一格到底存了什么？取数据的操作在系统层面到底长什么样？**

##### 存储形态：不是文本，是浮点向量

Engram 嵌入表里存的**不是**人类可读的事实（比如"巴黎是法国首都"这种文字），而是一组**浮点数向量**——和 word2vec / Transformer 的 token embedding 是同一类东西：

```
┌────────────────────────────────────────────────────────────────┐
│  子表 E₂,₁（以 2-gram + Head 1 为例）                          │
│                                                                │
│  在 DRAM 中的物理布局：一个连续的二维浮点数组                    │
│  float table[M][d]    （M ≈ 226万行, d ≈ 80列）                │
│                                                                │
│  索引    │       80 个 float32 数值（每行 320 字节）             │
│  ────────┼──────────────────────────────────────────            │
│  [0]     │  0.023, -0.871, 0.445, ..., 0.112                  │
│  [1]     │ -0.334,  0.209, 0.087, ..., -0.561                 │
│  [2]     │  0.756, -0.002, 0.933, ..., 0.418                  │
│  ...     │  ...                                                │
│  [198432]│  0.641, -0.378, 0.155, ..., 0.892   ← hash("of Wales") │
│  ...     │  ...                                                │
│  [M-1]   │ -0.127,  0.564, 0.231, ..., 0.047                 │
│                                                                │
│  每一行就是一条"记忆"——一个 80 维的语义坐标点                    │
│  这些数值是训练时通过梯度下降学出来的，不是人工写入的              │
└────────────────────────────────────────────────────────────────┘
```

关键理解：
* **向量是"学"出来的：** 如前文[哈希多头 ≠ Attention 多头](#哈希多头--attention-多头什么是学的什么不是)一节所述，哈希函数本身是固定的，但表中的向量参与反向传播。模型通过梯度下降自动决定在每个槽位存什么信息。
* **不可直接解读：** 你无法看着 `[0.641, -0.378, ...]` 就知道它编码了"威尔士王妃"。但当这个向量被 $\mathbf{W}_V$ 投影到模型的 2560 维隐藏空间后，它会在正确的维度上激活正确的语义特征。
* **哈希冲突是容忍的：** 不同的 N-gram 可能哈希到同一个槽位（冲突）。这就像两本不同的书被放到图书馆的同一个格子里——单个哈希头可能混淆，但 16 个头同时冲突的概率极低，多头拼接后仍能区分。

##### 检索操作：本质就是数组下标访问

你类比"堆内存 + 页查询"的直觉非常对，但 Engram 的检索比页查询还要简单——它就是**数组的下标直接寻址**，没有搜索、没有比较、没有树遍历：

```
【对比：几种"查知识"的方式】

  数据库查询（如 RAG）:
    SELECT embedding FROM knowledge WHERE text LIKE '%Princess of Wales%'
    → 需要索引遍历、字符串匹配、排序   → O(log N) ~ O(N)

  哈希表查询（如 Python dict）:
    dict[hash("Princess of Wales")]
    → 计算哈希 → 定位桶 → 处理冲突链   → 平均 O(1)，但有冲突处理开销

  Engram 查询:
    table[hash("Princess", "of", "Wales") % M]
    → 计算哈希 → 直接用整数做数组下标   → 严格 O(1)，无冲突处理（冲突由多头设计容忍）
```

在系统层面，一次 Engram 检索的完整操作是：

```
━━ 阶段 1: 计算地址（CPU 上，纳秒级）━━

  输入 N-gram: ("Princess", "of", "Wales")
  压缩 ID:     (38012, 7201, 41556)

  对 16 个 (阶, 头) 组合分别计算哈希：
    idx = (id₁ × A₁ ⊕ id₂ × A₂ ⊕ id₃ × A₃) mod M
                                          ↑
                            A₁, A₂, A₃ 是每个头的固定随机常数
                            ⊕ 是异或操作
                            M 是质数（表大小），取模避免越界

  得到 16 个整数索引: [198432, 67201, ..., 421098]

━━ 阶段 2: 从 DRAM 读取数据（PCIe DMA，微秒级）━━

  每个索引直接定位到 DRAM 中的物理地址：
    address = base_ptr + idx × 80 × sizeof(float)
            = base_ptr + 198432 × 320 字节
            = 某个确定的内存地址

  CPU 发起 16 次内存读取（或批量 DMA）：
    memcpy(gpu_buffer, address, 320 bytes)   ← 每次只读 320 字节

  通过 PCIe 总线传输到 GPU 显存

━━ 阶段 3: GPU 上拼接（纳秒级）━━

  16 个 80 维向量 → Concatenate → e_t ∈ ℝ^1280
  后续进入 W_K/W_V 投影和门控...
```

你的类比其实很准确——如果对应到程序员熟悉的概念：

```
┌──────────────┬─────────────────────────────────────────────┐
│  程序概念     │  Engram 中的对应                             │
├──────────────┼─────────────────────────────────────────────┤
│  堆（Heap）   │  DRAM 中的嵌入表就是一块巨大的堆内存          │
│              │  （连续分配的 float 数组）                    │
├──────────────┼─────────────────────────────────────────────┤
│  malloc/new  │  训练开始前分配：M × d × sizeof(float) 字节   │
│              │  Engram-27B 单层 ≈ 226万 × 1280 × 4 ≈ 11 GB │
├──────────────┼─────────────────────────────────────────────┤
│  指针 + 偏移  │  base_ptr + hash(n_gram) × entry_size       │
│  (寻址)      │  直接算出物理地址，无需搜索                    │
├──────────────┼─────────────────────────────────────────────┤
│  页缓存      │  论文的多级缓存层次：                         │
│  (Page Cache) │  热门 N-gram → GPU HBM（L1 缓存）           │
│              │  温热 N-gram → Host DRAM（L2 缓存）           │
│              │  冷门 N-gram → NVMe SSD（L3 缓存）            │
│              │  这部分和 OS 的页缓存思路完全一致！             │
├──────────────┼─────────────────────────────────────────────┤
│  DMA 传输    │  CPU 通过 PCIe 发起异步 DMA，                 │
│              │  把数据从 DRAM 搬到 GPU 显存                  │
│              │  （不走 CPU 缓存，直接总线传输）               │
└──────────────┴─────────────────────────────────────────────┘
```

一句话总结：**Engram 的查表本质上就是 `float_array[hash(n_gram)]`——堆上分配一个巨大的浮点数组，用哈希值当下标直接取。** 没有搜索引擎、没有树索引、没有键值比较。正是这种"笨办法"保证了严格 O(1) 和确定性地址（从而支持异步预取）。

##### 知识是怎么训练进去的？从空白到沉淀

一个关键区分：**训练时，嵌入表不在 DRAM 里——它在 GPU 上。** 只有推理部署时才迁移到 DRAM。整个知识沉淀的过程分三个阶段：

```
━━ 阶段 1: 初始化（训练开始前）━━

  226 万 × 80 维的浮点数组 → 随机初始化
  (就像一本字典，每一页写满了随机数字，完全没有意义)

  表被切片（Shard）分布到多块 GPU 上：
  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │  GPU 0   │  │  GPU 1   │  │  GPU 2   │  │  GPU 3   │
  │ 第0~56万 │  │第57~113万│  │第114~170万│ │第171~226万│
  │  行的向量 │  │  行的向量 │  │  行的向量 │  │  行的向量 │
  └─────────┘  └─────────┘  └─────────┘  └─────────┘
  (通过 All-to-All 通信，任何 GPU 都能访问任何切片)
```

```
━━ 阶段 2: 训练中（262B token，~50000 步）━━

  每一步训练的数据流：

  输入文本 "Diana, Princess of Wales was born in..."
      │
      ├── 前向传播（Forward）:
      │     hash("of", "Wales") → idx=198432
      │     取出 table[198432] = [0.023, -0.871, ...] (当前的随机向量)
      │     → 门控 → 卷积 → 注入残差流
      │     → Attention → MoE → ... → 预测下一个 token
      │     → 计算 Loss（预测错了 → Loss 大）
      │
      └── 反向传播（Backward）:
            梯度从 Loss 一路回传：
            Loss → ... → MoE → Attention → 门控 → W_V → table[198432]
                                                          ↑
                                          梯度告诉这个向量：
                                          "你应该往这个方向调整，
                                           才能让模型更好地预测"

  关键：只有被查到的那几行会收到梯度（稀疏更新）
  ┌────────────────────────────────────────────────────┐
  │  table[0]:       没被查到 → 梯度=0 → 不更新        │
  │  table[1]:       没被查到 → 梯度=0 → 不更新        │
  │  ...                                               │
  │  table[198432]:  被查到了 → 收到梯度 → Adam 更新!   │
  │  ...                                               │
  │  table[M-1]:     没被查到 → 梯度=0 → 不更新        │
  └────────────────────────────────────────────────────┘

  经过 262B token 的训练：
  高频 N-gram（"of the", "in a"）→ 被查了千百万次 → 向量充分优化
  低频 N-gram（罕见人名）→ 被查了几十次 → 向量大致有用
  从未出现的 N-gram → 向量仍是接近随机的噪声
```

```
━━ 阶段 3: 部署迁移（训练完成后）━━

  训练完毕，嵌入表的向量已经"长好了"

  GPU 上的切片 → 合并 → 整体搬到 DRAM
  ┌─────────┐┌─────────┐┌─────────┐┌─────────┐
  │  GPU 0   ││  GPU 1   ││  GPU 2   ││  GPU 3   │
  └────┬─────┘└────┬─────┘└────┬─────┘└────┬─────┘
       │           │           │           │
       └───────────┴───────────┴───────────┘
                       │
                 合并为完整的表
                       │
                       ▼
  ┌─────────────────────────────────────────────┐
  │  Host DRAM（推理服务器的主内存）               │
  │  完整的嵌入表：226万 × 80 × float32           │
  │                                              │
  │  此后表是只读的（推理不更新权重）               │
  │  推理时通过 PCIe 异步预取到 GPU               │
  └─────────────────────────────────────────────┘
```

##### 等等——所有模块一起训练，梯度不会"分不清"吗？

⚠️ 重要澄清：**Engram-27B 不是在冻结的模型上微调 Engram，而是所有模块（Attention、MoE/FFN、Engram 嵌入表、门控 $\mathbf{W}_K$/$\mathbf{W}_V$、Conv）从零开始、在同一批 262B token 上联合训练。**

那问题来了：反向传播时，梯度怎么知道"这个预测错误该怪 Engram 查表不准，还是该怪 Attention 没关注到关键位置，还是该怪 MoE 推理能力不够"？

答案是：**不需要"判断"，链式法则会把梯度同时分发给每一个参与过前向计算的参数。**

```
前向传播（一个 token 的计算路径）：

  输入 → Engram(hash查表 → 门控 → Conv) → (+残差)
                                              │
                                         Attention
                                              │
                                          MoE/FFN
                                              │
                                         预测 → Loss = 2.3

反向传播（梯度沿所有路径同时回流）：

  Loss = 2.3
      │
      ├──→ MoE/FFN 的每个专家权重     ← 收到梯度 → 用 Muon 更新
      │
      ├──→ Attention 的 Q/K/V 权重    ← 收到梯度 → 用 Muon 更新
      │
      ├──→ 门控 W_K, W_V 矩阵         ← 收到梯度 → 用 Muon 更新
      │
      └──→ Engram table[198432]       ← 收到梯度 → 用 Adam 更新
           (只有被查到的这一行)                     (5倍学习率)

  每个参数收到的梯度 = ∂Loss/∂该参数
  就是"如果只微调你一个人，Loss 会怎么变"的方向
```

这和你熟悉的 Transformer 训练完全一样——在标准 Transformer 里，你也不需要手动判断"这次预测错了是 Attention 的问题还是 FFN 的问题"，反向传播的链式法则自动把梯度按照每个参数对 Loss 的贡献大小分配下去。Engram 只是多了一条从 DRAM 查表的路径，但它同样是可微分的计算图的一部分。

**那模型怎么自动学会"Engram 存事实、FFN 做推理"这种分工？**

这不是人为指定的，而是**架构的归纳偏置（Inductive Bias）**自然涌现出来的：

```
┌─────────────────────────────────────────────────────────────┐
│                    架构特性决定了能力边界                      │
├────────────────┬────────────────────────────────────────────┤
│  Engram 查找表  │ 输入 = 固定的 N-gram 哈希                  │
│                │ → 只能捕获局部的、静态的、与上下文无关的模式  │
│                │ → 天然适合: "Princess of Wales", "of the"   │
│                │ → 天然不适合: 需要长距离推理的任务            │
├────────────────┼────────────────────────────────────────────┤
│  Attention     │ 输入 = 所有位置的 Q/K/V                     │
│                │ → 能建模任意长距离依赖                       │
│                │ → 天然适合: 指代消解、远距离逻辑关联          │
├────────────────┼────────────────────────────────────────────┤
│  MoE/FFN       │ 输入 = 当前位置的隐藏状态                    │
│                │ → 强大的非线性变换能力                       │
│                │ → 天然适合: 复杂推理、数学计算、代码逻辑      │
└────────────────┴────────────────────────────────────────────┘

训练过程中自然发生的事情：

  "巴黎是法国首都" 这类知识：
  → Engram 查表就能搞定（O(1)，梯度让嵌入向量存好这条知识）
  → Attention 和 FFN 发现"不用我出力"→ 它们的梯度在这类任务上变小
  → 结果：FFN 的容量被释放出来做更难的事

  "如果 A > B 且 B > C，那么 A > C" 这类推理：
  → Engram 的哈希查表根本无法表达这种逻辑（它没有全局上下文）
  → 门控 α_t ≈ 0，自动屏蔽 Engram 的输出
  → Attention + FFN 承担全部工作
```

这就像给一个学生**同时发一本词典和一个计算器**——你不需要告诉他"查事实用词典、做数学用计算器"，训练（考试）过程中他自然会摸索出最高效的分工。论文中的 Sensitivity Analysis 实验也验证了这一点：**关闭 Engram 后，事实类任务暴跌到 29%，但阅读理解仅下降到 93%**——说明模型确实学会了让 Engram 主攻事实、骨干主攻推理。

![Figure 6: Engram 消融后的保留性能。事实知识严重依赖 Engram（TriviaQA 仅保留 29%），而阅读理解主要由骨干网络支撑（C3 保留 93%）。](https://arxiv.org/html/2601.07372v1/x6.png)

训练的几个重要工程细节：
* **优化器：** 嵌入表用 Adam（不是骨干网络用的 Muon），学习率是骨干的 **5 倍**——因为每个向量只有被哈希命中时才能更新，需要更大步长来弥补更新频率低的问题。
* **无权重衰减：** 嵌入表不做 weight decay，防止低频 N-gram 的知识被逐渐"遗忘"回零。
* **卷积零初始化：** 门控后面的 Conv1D 权重初始化为全零，确保训练刚开始时 Engram 的输出恒等于零（$\mathbf{Y}=0$），不会破坏预训练骨干的稳定性。随着训练推进，卷积权重逐渐学出非零值，Engram 的贡献平滑地从 0 增长。

#### C. 门控机制详解：一个微型 Attention

查表拿到 $e_t$ 之后，核心问题是：这条记忆**要不要用、用多少**？

Engram 用了一个非常精巧的设计——**类似 Attention 的 QKV 机制**，但输出的是一个标量门控值而非向量：

```
当前隐藏状态 h_t ∈ ℝ^2560              从 DRAM 查出的记忆 e_t ∈ ℝ^1280
(来自上一层的输出，                      (N-gram 哈希查表结果)
 已经过 Attention                                │
 聚合了全局上下文)                         ┌──────┴──────┐
       │                                  │             │
       │  作为 Query                     W_K           W_V
       │                              (2560×1280)   (2560×1280)
       │                                  │             │
       │                                  ▼             ▼
       │                           k_t ∈ ℝ^2560   v_t ∈ ℝ^2560
       │                            (记忆的"签名")   (记忆的"内容")
       │                                  │             │
       ▼                                  ▼             │
   RMSNorm(h_t)                     RMSNorm(k_t)        │
       │                                  │             │
       └───── 点积 / √d ────────────────┘             │
                    │                                   │
                  σ(·) sigmoid                          │
                    │                                   │
              α_t ∈ (0, 1)                              │
              ┌─ 标量门控值 ─┐                          │
              │              │                          │
              │   α_t ≈ 1:  记忆与语境匹配 → 大量采纳  │
              │   α_t ≈ 0:  记忆与语境矛盾 → 自动忽略  │
              └──────────────┘                          │
                    │                                   │
                    └───────── α_t × v_t ──────────────┘
                                   │
                            ṽ_t = α_t · v_t
                            (门控后的值向量)

  ⚠ 注意 α_t 是单个标量（一个数字），不是向量。
  v_t 的 2560 个维度全部乘以同一个 α_t——
  这是对记忆整体的"音量旋钮"，不是逐维度的独立开关。
  图中 ⊗ 符号就是这步：标量 × 向量。

                                   │
                             ┌─────┴─────┐
                             │ RMSNorm   │
                             │     ↓     │
                             │ Conv1D    │  ← 因果深度卷积 (kernel=4, dilation=3)
                             │ (depthwise│     扩展感受野，增强非线性
                             │  causal)  │
                             │     ↓     │
                             │   SiLU    │  ← 激活函数
                             │     ↓     │
                             │ (+) ← ṽ_t │  ← 残差连接
                             └─────┬─────┘
                                   │
                                   ▼
                            Y_t ∈ ℝ^2560
                           (Engram 最终输出)
```

这个公式的数学表达：

$$\alpha_t = \sigma\!\left(\frac{\text{RMSNorm}(\mathbf{h}_t)^\top \, \text{RMSNorm}(\mathbf{W}_K \mathbf{e}_t)}{\sqrt{d}}\right)$$
$$\tilde{\mathbf{v}}_t = \alpha_t \cdot (\mathbf{W}_V \mathbf{e}_t)$$
$$\mathbf{Y} = \text{SiLU}\!\left(\text{Conv1D}(\text{RMSNorm}(\tilde{\mathbf{V}}))\right) + \tilde{\mathbf{V}}$$

##### 维度对齐：记忆是怎么"拼"回输入序列的？

上面的图展示了单个位置 $t$ 的计算。但实际输入是一个长度为 $T$ 的序列，Engram 的输出怎么和它对齐？

关键：**Engram 在每个 token 位置独立运算，不存在长度不对称问题。** 下图展示了完整的序列维度数据流：

```
输入序列 H^(l) ∈ ℝ^{T × 2560}（T 个 token，每个 2560 维）

  位置 1: h₁    位置 2: h₂    位置 3: h₃   ...  位置 T: h_T
   │              │              │                  │
   ▼              ▼              ▼                  ▼
  hash(1-gram)  hash(2-gram)  hash(3-gram)       hash(N-gram)
   │              │              │                  │
   ▼              ▼              ▼                  ▼
  e₁ ∈ ℝ^1280  e₂ ∈ ℝ^1280  e₃ ∈ ℝ^1280       e_T ∈ ℝ^1280
   │              │              │                  │
   ▼              ▼              ▼                  ▼
  W_V 投影       W_V 投影       W_V 投影           W_V 投影
  (1280→2560)   (1280→2560)   (1280→2560)        (1280→2560)
   │              │              │                  │
   ▼              ▼              ▼                  ▼
  v₁ ∈ ℝ^2560  v₂ ∈ ℝ^2560  v₃ ∈ ℝ^2560       v_T ∈ ℝ^2560
   │              │              │                  │
   ▼              ▼              ▼                  ▼
  α₁·v₁         α₂·v₂         α₃·v₃              α_T·v_T
  (门控)         (门控)         (门控)              (门控)
   │              │              │                  │
   └──────────────┴──────────────┴──────────────────┘
                              │
                       堆叠为 Ṽ ∈ ℝ^{T × 2560}
                       （和 H^(l) 形状完全一致！）
                              │
                       Conv1D（沿时间轴 T 做因果卷积）
                       （不改变形状，只做相邻位置间的局部混合）
                              │
                       Y ∈ ℝ^{T × 2560}
                              │
                              ▼
                 H^(l) = H^(l) + Y    ← 逐位置相加（残差连接）
                 ┌──────────────────────────────────────┐
                 │  h₁+Y₁,  h₂+Y₂,  h₃+Y₃, ..., h_T+Y_T │
                 │  每个位置的隐藏状态被对应的记忆"增强"了  │
                 └──────────────────────────────────────┘
                              │
                              ▼
                    送入 Attention 作为输入
```

所以回答你的问题：

1. **"长度不对称"？** — 不存在。每个 token 位置独立触发一次 N-gram 查表，T 个位置产生 T 个输出，和输入序列等长。
2. **"维度不对称"？** — 由 $\mathbf{W}_V$（2560×1280）投影解决：查表得到的 1280 维向量被线性映射为 2560 维，与隐藏状态维度一致。
3. **"卷积做了什么"？** — Conv1D 不改变形状（same-padding，因果），它只是沿时间轴让**相邻位置的 Engram 输出之间做局部混合**（kernel=4, dilation=3 意味着往回看最多 9 个位置），增加一点非线性和局部感知。
4. **"怎么拼到输入的"？** — **直接逐位置相加**（残差连接），不是拼接（concatenate）。$h_t$ 被 $Y_t$ "增强"后，就成为 Attention 的输入——Attention 看到的每个位置的向量里，已经包含了 Engram 查表注入的静态知识。

##### 多分支集成（mHC, M=4）— 这是骨干网络的改进，不是 Engram 的

先说清楚这一节在讲什么：**mHC 和 N-gram、层数都无关**，它是 DeepSeek 在另一篇论文《[mHC: Manifold-Constrained Hyper-Connections](https://arxiv.org/abs/2512.24880)》（2025.12）中提出的**骨干网络架构改进**，Engram 论文只是拿它当默认骨干。

**mHC 解决的问题：** 标准 Transformer 的残差连接（`output = x + F(x)`）只有一条信息通道，容量有限。前人提出了 Hyper-Connections（HC）把残差流扩展为多条并行分支来增大容量，但破坏了恒等映射（identity mapping）特性，导致训练不稳定、无法 Scale Up。mHC 通过将连接权重约束到**双随机矩阵流形**（Sinkhorn-Knopp 算法）上，恢复了恒等映射特性，从而同时获得多分支的表达力和单残差流的训练稳定性。

**实际效果：** mHC 已被用于 DeepSeek 的多个生产级模型（包括 Engram-27B/40B），实验证明在同等参数量下有显著的性能提升和更好的扩展性，额外计算开销可忽略。

**标准 Transformer 的残差流 vs mHC 的多分支残差流：**

```
【标准 Transformer —— 单条残差流】

  Token Embedding（每个 token 1 个向量）
         │
         ▼
       h_t ∈ ℝ^2560       ← 每个位置始终只有 1 个向量
         │
     [Layer 1: Attention → FFN]
         │
     [Layer 2: Attention → FFN]
         │
       ...30 层...
         │
     [LM Head] → 预测下一个 token


【mHC (M=4) —— 4 条并行残差流】

  Token Embedding（每个 token 仍然只有 1 个向量）
         │
         ▼
       h_t ∈ ℝ^2560       ← 输入时确实只有 1 个
         │
    ┌────┴────┐
    │ 扩展映射 │  ← 可学习的线性变换，把 1 个向量扩展为 4 个
    └────┬────┘
         │
    ┌────┼────┬────┐
    ▼    ▼    ▼    ▼
  h^(1) h^(2) h^(3) h^(4)   ← 现在每个位置有 4 个隐藏向量
    │    │    │    │
    │  [Layer 1: Attention → FFN]  ← 4 条流并行通过每一层
    │    │    │    │
    │  [Layer 2: Attention → FFN]
    │    │    │    │
    │  ...30 层...
    │    │    │    │
    └────┼────┴────┘
    ┌────┴────┐
    │ 收缩映射 │  ← 把 4 个向量合并回 1 个
    └────┬────┘
         │
         ▼
       h_t ∈ ℝ^2560       ← 输出时又变回 1 个
         │
     [LM Head] → 预测下一个 token
```

所以关键点是：**embedding 确实只有 1 个，但进入 Transformer 层之前会被线性扩展为 4 份，最后输出前再收缩回 1 份。** 中间的 30 层都在 4 条并行流上运算。这类似于多头 Attention 把 Q/K/V 拆成多个头——只是 mHC 在残差流层面做了类似的事情，让不同"车道"可以承载不同维度的表征。这是骨干网络层面的优化，和 Engram 的查表逻辑无关。

**Engram 需要适配这 4 条分支：**

既然骨干是 4 分支的，那 Engram 查表出来的记忆该怎么分配给这 4 条流？论文的设计是——**查表共享，门控独立**：

```
             查表结果 e_t ∈ ℝ^1280（4 条分支共用同一个查表结果）
                          │
                    ┌─────┴─────┐
                    │           │
                   W_V         W_K^(1)  W_K^(2)  W_K^(3)  W_K^(4)
                 (共享1个)      (每条分支各自 1 个)
                    │           │        │        │        │
                    ▼           ▼        ▼        ▼        ▼
               v_t (共享)    k_t^(1)  k_t^(2)  k_t^(3)  k_t^(4)
                    │           │        │        │        │
                    │           └── 各自与本分支的 h_t^(m) 做点积 ──┐
                    │                                              │
                    │         α_t^(1)   α_t^(2)  α_t^(3)  α_t^(4) │
                    │         (0.9)     (0.3)    (0.1)    (0.7)    │
                    │           │        │        │        │       │
                    └───────→ α^(m) × v_t（每条分支独立决定采纳多少）│
                              │        │        │        │        │
                              ▼        ▼        ▼        ▼        │
                          分支1:    分支2:    分支3:   分支4:      │
                          大量     少量      几乎     较多        │
                          采纳     采纳      忽略     采纳        │
```

设计思路：
* **共享嵌入表 + 共享 $\mathbf{W}_V$**：4 条分支从同一条记忆中提取同一份"内容"（$\mathbf{v}_t$），避免参数浪费（不需要 4 套嵌入表）
* **独立 $\mathbf{W}_K^{(m)}$**：每条分支用自己的 Key 投影计算门控值 $\alpha_t^{(m)}$，可以独立决定"要不要采纳这条记忆"
* **效果**：同一条 N-gram 记忆，分支 1 可能大量采纳（$\alpha^{(1)}=0.9$），分支 3 可能忽略（$\alpha^{(3)}=0.1$）——各取所需
* **工程优化**：1 个 $\mathbf{W}_V$ + 4 个 $\mathbf{W}_K^{(m)}$ 可以合并为一次 FP8 矩阵乘法，充分利用 GPU 算力

如果你不关注 mHC 这种骨干改进，可以把这一节理解为：**标准单残差流的 Engram（上面 C 节讲的）就是完整的核心设计，mHC 只是工程上的进一步适配。**

#### D. 在 Transformer Block 中的真实处理顺序

⚠️ **重要澄清**：Engram 模块**不是**与 FFN/MoE 并行执行，而是位于整个 Attention + MoE **之前**。这是论文原文的明确表述：$\mathbf{H}^{(\ell)} \leftarrow \mathbf{H}^{(\ell)} + \mathbf{Y}$，然后才进入标准的 Attention 和 MoE。

```
┌────────────────────────────────────────────────────────────────┐
│            Engram 增强 Transformer Block（Layer 2 或 15）        │
│                                                                │
│  输入 H^(l)                                                    │
│      │                                                         │
│      │        ┌──────────────────────────────────┐             │
│      ├──────→ │  ① Engram 模块                    │             │
│      │        │  哈希查 DRAM → W_K/W_V 投影       │             │
│      │        │  → 门控 α_t → 卷积精炼 → Y        │             │
│      │        └──────────────┬───────────────────┘             │
│      │                       │                                 │
│      └───────→ (+) ◄─────────┘   H^(l) = H^(l) + Y            │
│                │                                               │
│                ▼                                               │
│         ② Attention（标准 MLA，32 头）                          │
│                │          ↑ 现在 Attention 的输入已经包含了      │
│                │            Engram 查表注入的静态知识            │
│                ▼                                               │
│         ③ MoE / FFN（55 个路由专家，top-6 激活）                │
│                │                                               │
│                ▼                                               │
│  输出 H^(l+1)                                                  │
└────────────────────────────────────────────────────────────────┘

对比：普通 Transformer Block（30 层中的其他 28 层）

  输入 H^(l) → Attention → MoE/FFN → 输出 H^(l+1)
  （无 Engram，全部在 GPU HBM 内完成）
```

这个位置选择有两个约束条件的**联合优化**：

* **模型角度（越早越好）：** 论文的 Layer Sweep 实验证明 Layer 2 是单层最优。越早注入，越能在 Attention 之前就把静态知识"塞"进残差流，让后续的 Attention 和 MoE 不用浪费计算去重构实体（如 "Diana, Princess of Wales" 原本需要 6 层逐步拼装，Engram 在 Layer 2 一次查表搞定）。
* **系统角度（不能太早）：** Layer 1 太早了——门控机制需要 $\mathbf{h}_t$ 已经聚合了一定的上下文信息（至少过一轮 Attention）。同时，Layer 1 的计算时间是 DRAM → GPU 数据搬运通过 PCIe 的时间窗口。

#### E. 端到端完整数据流

最后，把所有模块串起来，看一个 token 从输入到 Engram 增强的完整旅程：

```
输入 token: "Diana, Princess of Wales" 中的最后一个 token "Wales"

━━ 步骤 1: Tokenizer 压缩 ━━
  "Wales" → token_id = 42195 → 压缩后 id = 38012
  (合并大小写变体："Wales"/"wales"/"WALES" → 同一个 ID)

━━ 步骤 2: 构造后缀 N-gram ━━
  2-gram: (of', Wales')
  3-gram: (Princess', of', Wales')

━━ 步骤 3: 多头哈希查表（从 DRAM 中取数据）━━
  2-gram 经 8 个哈希头 → 8 个地址 → 从 DRAM 取出 8 个子向量
  3-gram 经 8 个哈希头 → 8 个地址 → 从 DRAM 取出 8 个子向量
  Concatenate → e_t ∈ ℝ^1280（编码了 "Princess of Wales" 的静态知识）

━━ 步骤 4: 上下文感知门控 ━━
  h_t = Layer 1 输出的隐藏状态（已包含 "Diana" 等上下文）
  k_t = W_K · e_t       → 记忆的"签名"  ∈ ℝ^2560
  v_t = W_V · e_t       → 记忆的"内容"  ∈ ℝ^2560
  α_t = sigmoid(RMSNorm(h_t)ᵀ · RMSNorm(k_t) / √2560)
      = 0.92             → "Princess of Wales" 与上下文高度匹配

━━ 步骤 5: 门控 + 卷积精炼 ━━
  ṽ_t = 0.92 × v_t                → 几乎全量采纳这条记忆
  Y_t = SiLU(Conv1D(RMSNorm(ṽ_t))) + ṽ_t   → 卷积扩展感受野

━━ 步骤 6: 残差注入 ━━
  H^(2) = H^(2) + Y_t              → 注入到 Layer 2 的残差流

━━ 后续：标准 Transformer 继续 ━━
  → Attention（输入已包含 Engram 注入的实体知识,
              现在可以专注于长距离依赖和推理）
  → MoE/FFN （55 个专家处理复杂逻辑）
  → ... 后续 Layer 3-30 ...
```

这就是为什么论文中发现，加了 Engram 后 **Layer 5 的表征质量等价于纯 MoE 模型 Layer 12**——Engram 在浅层就把"死记硬背"的工作做完了，相当于给模型白送了好几层的有效深度。

### 3. U 型比例定律 (U-shaped Scaling Law)与权重分配

DeepSeek 重新定义了模型性能与参数分配的关系，发现了一个反直觉的 **U 型曲线**。

*   **占用了谁的权重？** 主要是 **FFN**。
    *   正如我们之前讨论的，FFN 占据了模型约 2/3 的参数，主要负责存储静态知识。
    *   Engram 的本质是用廉价的**存储参数（Memory Parameters，存在 DRAM 里）**去替代昂贵的**计算参数（Compute Parameters，即 FFN 权重，存在 HBM 里）**。

![Figure 3: 稀疏分配与 Engram 缩放。左图：不同分配比例 ρ 下的验证损失（两个计算预算均呈 U 型，混合分配优于纯 MoE）。右图：无限存储体制下的缩放行为（验证损失与嵌入数量呈对数线性关系）。](https://arxiv.org/html/2601.07372v1/x3.png)

*   **U 型发现：** 在**固定总预算**（计算成本 + 存储成本）的情况下：
    *   **纯计算（100% FFN）：** 成本高，且容易遗忘长尾知识。
    *   **纯存储（100% N-gram）：** 泛化能力差，只能复读。
    *   **混合最优（~75% FFN + ~25% Engram）：** 曲线的**最低点（Sweet Spot）**。这意味着将约 25% 的 FFN 参数"割让"给 Engram 存储表，能获得最佳的 Scaling 效果。

这背后的统计学原理是 **Zipf 定律**：语言中 20% 的高频静态模式（适合查表）覆盖了 80% 的日常表达，而剩下的长尾逻辑（需要推理）则交给昂贵的 FFN 处理。

## 还有更好的解决方案吗？

Engram 是目前最前沿的方向之一，但确实存在其他竞争路线：

* **RAG（检索增强生成）：** 这种是在模型外面挂个搜索引擎。优点是知识可以实时更新，缺点是速度慢、推理成本高，且无法改变模型底座的效率。
* **K-NN LM：** 类似思路，但通常计算成本巨大，难以在超大规模预训练中使用。
* **Infini-transformer：** 谷歌提出的方案，试图通过特殊的注意力机制实现无限长记忆，但工程实现难度极大。

**Engram 的优势在于"工程美学"：** 它非常符合硬件（CPU/GPU 分层存储）的现状，且能直接集成进预训练过程中。

要深入理解 DeepSeek Engram，我们需要把这些硬核的技术名词拆解成直观的模型。Engram 的核心逻辑其实就是：**既然模型记不住那么多东西，那就给它一套高效的"查词典"系统。**

## 关键词解析

### 1. 存储"稀疏化" (Storage Sparsity)

这是相对于**计算稀疏化（MoE）**提出的概念。三种模式的对比：

```
【稠密模型（Dense）— 全量计算，全量存储】

  输入 token → [所有神经元全部激活] → 输出
                 ↑
            所有权重都在 HBM 里，全部参与计算

【MoE 模型 — 计算稀疏，但存储仍然全量】

  输入 token → [路由器] → 只激活 2/8 个专家
                           ↑
            所有 8 个专家的权重仍然在 HBM 里（只是没被计算）

【Engram 模型 — 计算稀疏 + 存储稀疏】

  输入 token → [路由器] → 只激活 2/8 个专家（计算稀疏）
       │
       └→ [哈希] → 从 DRAM 中只取出对应的几条向量（存储稀疏）
                    ↑
            数十亿条向量在 DRAM 里，99.9% 不动
```

* **通俗理解：** 以前是考试前把整本书背下来（稠密），现在是允许你带一本巨大的字典进考场，但你每次只翻开那一页（稀疏）。

### 2. 确定性寻址和异步预取 (Deterministic Addressing & Asynchronous Prefetching)

![Figure 2: Engram 系统实现。(a) 训练阶段：嵌入表分片到多块 GPU，通过 All-to-All 通信检索活跃行。(b) 推理阶段：嵌入表卸载到主机内存，利用确定性寻址异步预取，与前层计算重叠隐藏延迟。](https://arxiv.org/html/2601.07372v1/x2.png)

这是解决**"查找表在 DRAM 里，GPU 读它会不会太慢？"**的工程方案。

* **确定性寻址：** Engram 使用固定的哈希函数来计算知识存放在 DRAM 的哪个地址。模型看到"苹果"两个字，不需要搜索，直接根据公式算出地址，复杂度是 O(1)。
* **异步预取：** DRAM 读取速度（~500 GB/s）远慢于 HBM（~3 TB/s）。为了不让 GPU 闲着等数据，Engram 采用了**逐层流水线**——在计算当前层的同时，提前把下一层需要的数据从 DRAM 搬到 GPU：

```
时间 ──────────────────────────────────────────────────────→

GPU 计算:  ┌─Layer 1─┐  ┌─Layer 2─────┐  ┌─Layer 3─────┐
           │Attn+FFN │  │Attn+FFN     │  │Attn+FFN     │
           │(无Engram)│  │+Engram查表  │  │+Engram查表  │
           └─────────┘  └─────────────┘  └─────────────┘

PCIe 搬运: ──────────── ┌搬Layer2数据┐  ┌搬Layer3数据┐
(DRAM→GPU)              └───────────┘  └───────────┘
                         ↑                ↑
                    计算Layer1时         计算Layer2时
                    就开始搬Layer2      就开始搬Layer3
                    的查表数据          的查表数据

                    → 等到要用时，数据已经在 GPU 里了！
```

* 当模型在计算第 1 层时，它已经算好了第 2 层需要查哪些知识的地址。
* 通过 PCIe 总线**异步**地把数据从 DRAM 搬往 GPU。
* 等第 2 层开算时，数据已经在那等着了，几乎实现了"零成本"查询。

### 3. U 型比例定律 (U-shaped Scaling Law)

这是该论文最重大的理论发现。DeepSeek 重新定义了模型性能与参数分配的关系。

* **传统认识：** 增加参数就能变强，曲线是单调下降的。
* **Engram 的发现：** 在**固定总参数**（计算参数 + 存储参数）的情况下，如果把存储参数比例从 0% 逐渐增加，模型损失（Loss）会先下降后上升。
* **结论：** 存在一个**最优配比**（通常在 20%-25% 左右）。这意味着如果你有 100 亿参数的预算，全给计算（模型权重）不是最优的；给 80 亿计算，20 亿做 Engram 存储，效果最好。这个最优点的凹陷处就是"U 型"的底部。
* **深层原因（Zipf 定律）：** 这个比例并非偶然，而是对应了自然语言和代码中的**Zipf 分布（长尾分布）**。
    * 语言中少量的**高频静态模式**（如常见短语、固定搭配）占据了大部分出现频率，适合用 Engram 这种低成本的"查表"方式**死记硬背**。
    * 而剩下的**长尾复杂逻辑**（需要推理的部分）则交给昂贵的 MoE 计算参数去处理。
    * 20% 的存储参数解决了 80% 的常见模式查找，这就是"好钢用在刀刃上"的统计学体现。

### 4. 多头哈希 (Multi-Head Hashing)

灵感来源于 Transformer 的"多头注意力机制"（具体的张量形状和计算细节见上文 [2.5 B 节](#B-多头哈希网络的内部结构)）。

* **问题：** 哈希会有"冲突"。比如"苹果"既是水果也是公司，如果只用一个哈希函数，可能会把这两个含义混在一起。
* **解决方案：** 像 Attention 有 8 个头一样，Engram 也用 K=8 个独立的乘法-异或哈希函数，再乘以 N-gram 阶数（{2,3}），共 16 个子表。
    * 不同头+不同阶的哈希函数从不同维度定位知识
    * 每个子表返回一个子向量，16 个拼接成完整的记忆向量 $e_t$
* **效果：** 通过多个维度进行查询，模型能更精准地从海量表里勾勒出它现在最需要的那部分知识，极大地减少了查询误差。

### 5. 上下文感知门控 (Context-Aware Gating)

防止模型"查字典查傻了"（具体的 QKV 机制和公式见上文 [2.5 C 节](#C-门控机制详解一个微型-Attention)）。

* **本质：** 一个微型 Attention 机制。当前隐藏状态 $\mathbf{h}_t$ 作为 Query，查表结果 $\mathbf{e}_t$ 经过 $\mathbf{W}_K$/$\mathbf{W}_V$ 投影生成 Key 和 Value。
* **门控值：** $\alpha_t = \sigma(\text{RMSNorm}(\mathbf{h}_t)^\top \text{RMSNorm}(\mathbf{k}_t) / \sqrt{d})$，是一个 (0,1) 之间的**标量**。
* **应用逻辑：**
    * 语境匹配 → $\alpha_t \to 1$ → 大量采纳记忆
    * 语境矛盾 → $\alpha_t \to 0$ → 自动忽略，模型靠自己的推理
* **融合方式：** Engram 的输出通过**残差连接**加到 Attention 之前（不是加到 FFN 输出上）：$\mathbf{H}^{(\ell)} \leftarrow \mathbf{H}^{(\ell)} + \mathbf{Y}$，然后才进入 Attention → MoE/FFN。

下图是论文中对 Engram-27B 门控值 $\alpha_t$ 的实际可视化。可以看到门控在完成静态模式（如多 token 实体名 "Alexander the Great"、习语 "四大发明"）时强烈激活（红色），而在普通 token 上保持低值：

![Figure 7: 门控机制可视化。热力图强度对应门控标量 α_t ∈ [0,1]，深红色表示强激活。高激活出现在静态模式完成时（如命名实体、固定搭配），说明 Engram 成功识别并检索了定型化的语言依赖。](https://arxiv.org/html/2601.07372v1/x7.png)

## 还有哪些值得深挖的洞察？

除了 Zipf 定律，Engram 还有几个非常反直觉但深刻的架构洞察，值得深入思考：

### 1. 位置的哲学：记忆是"低级"认知

论文中一个有趣的细节是 Engram 模块的最佳插入位置通常在模型的**浅层（如第 2-4 层）**。

* **反直觉：** 我们通常认为"知识"是高级的，应该在深层处理。
* **深层解释：** 事实回忆其实是一种**"反射"**，而非**"思考"**。
    * 当你看到"白日依山"，大脑瞬间弹出"尽"，这不需要复杂的逻辑推理。
    * Engram 在浅层把这些"死记硬背"的工作做完了，**解放了深层网络**。
    * 结果：深层的 Attention 和 FFN 不再需要浪费算力去重构"巴黎在法国"这种简单事实，而是可以全心全意进行复杂的逻辑推演（Reasoning）。这也解释了为什么加了 Engram 后，模型的**推理能力**（如数学、代码）反而提升了。

#### 实验验证：LogitLens 和 CKA 分析

上面的"解放深层"不只是直觉，论文用了两个可解释性分析工具来严格验证。

**工具 1: LogitLens（逐层预测收敛分析）**

LogitLens 的原理是：把模型每一层的中间隐藏状态直接接到最终的 LM Head（输出层）上，看它能不能"提前"做出正确预测。用 KL 散度衡量中间层输出分布与最终层输出分布的距离——越小说明这一层就已经"想明白了"。

**工具 2: CKA（跨模型表征对齐分析）**

CKA（Centered Kernel Alignment）衡量的是两组表征的**结构相似度**——不看绝对数值，而是看"信息的组织方式是否一致"。论文用它来对比 Engram 模型和 MoE 模型在不同层的表征：

$$\text{CKA}(K, L) = \frac{\text{HSIC}(K, L)}{\sqrt{\text{HSIC}(K, K) \cdot \text{HSIC}(L, L)}}$$

其中 $K = XX^\top$, $L = YY^\top$ 是两组表征的 Gram 矩阵，HSIC 是 Hilbert-Schmidt 独立性准则。

**论文原图（Figure 4）：**

![Figure 4: LogitLens KL 散度分析（左）和 CKA 表征对齐热力图（中、右）](https://arxiv.org/html/2601.07372v1/x4.png)

> 图 (a): **LogitLens 逐层 KL 散度**。Engram 模型（红/橙线）在浅层的 KL 散度显著低于 MoE 基线（蓝线），说明 Engram 更早完成特征组装。差距在 Layer 2-5 最大，深层趋于一致。
>
> 图 (b)(c): **CKA 相似度热力图**。白色虚线为"软对齐曲线"——如果 Engram 和 MoE 每层表征相同，这条线应该贴着对角线。但实际曲线明显**偏移到对角线上方**，意味着 Engram 的第 $j$ 层表征等价于 MoE 的第 $j + \Delta$ 层（$\Delta > 0$）。最显著的发现：**Engram Layer 5 ≈ MoE Layer 12**。

结论：Engram 通过在浅层查表注入事实知识，相当于给模型"白送"了约 7 层的有效深度，让真正用于复杂推理的"有效网络"更深了。

这两个分析工具的一致结论：**Engram 不只是一个"查字典"的辅助模块，它从根本上改变了模型的信息处理流程——把浅层从"死记硬背"中解放出来，让整个网络的有效深度大幅增加。**

### 2. 内化的 RAG：模糊了"模型"与"数据"的界限

Engram 经常被拿来和 RAG（检索增强生成）做对比，但它们有一个本质区别：**可微性（Differentiability）**。

* **RAG 是"外挂"：** 数据库是冻结的，检索器通常是独立训练的。模型发现检索错了，很难直接通过反向传播去"修正"数据库的索引策略。
* **Engram 是"器官"：** 它是**端到端训练（End-to-End Trained）**的。
    * Engram 的查表过程、门控机制、甚至存储的向量（在训练时），都是参与梯度下降的。
    * 这意味着模型不仅学会了"怎么查"，还学会了"存什么"以及"什么时候信，什么时候不信"。这是一个**有自我意识的知识库**。

### 3. 信息熵的极限压缩

Engram 在构建查找表时做了一个 **Tokenizer Compression（词表压缩）** 的操作，比如把 "The", "the", " THE" 映射到同一个键值。

* **表面看：** 这是一个节省空间的工程技巧。
* **深层看：** 这是对**语义空间**的再定义。
    * 传统 Transformer 对大小写敏感，是因为它们是不同的 token ID。
    * Engram 强迫模型在存储层面进行**语义归一化**。这实际上是在降低信息的**熵（Entropy）**，让有限的存储空间（RAM）能装下更高密度的"概念"，而非"字符串"。这是对 Scaling Law 中数据质量的一种隐式提升。

---

### 总结

Engram 的本质是 **"知识的降本增效"**：

1. **存储稀疏化**让知识不再占用昂贵的 GPU 神经元。
2. **多头哈希和确定性寻址**让找知识变得极快。
3. **门控**确保知识用得对。
4. **U 型定律**指导我们如何科学地分配大脑容量。
