*论文发布时间：2023-03-31*

KAUST（沙特阿卜杜拉国王科技大学）发布的 **CAMEL**（论文标题为《[CAMEL: Communicative Agents for "Mind" Exploration of Large Language Model Society](https://arxiv.org/abs/2303.17760)》，NeurIPS 2023）是多 Agent 协作领域的早期奠基性工作之一。
简单来说，这篇论文提出了一种**"角色扮演"（Role-Playing）框架**，通过精心设计的**"盗梦"提示（Inception Prompting）**让两个 LLM Agent 自主分工协作完成任务——一个扮演"用户"负责拆解任务和下达指令，一个扮演"助手"负责执行指令和输出方案——**整个过程几乎不需要人类介入**。

## CAMEL 解决了什么问题？

**核心问题：LLM 的成功严重依赖人类的持续引导，如何让 Agent 自主协作而不需要人类手把手指挥？**

2023 年初，ChatGPT 等对话模型已经展示了强大的任务求解能力。但有一个根本限制——**它们需要人类不断地提供精准的 prompt 来引导对话方向**：

```
【人类引导 LLM 的困境】

  用户: "帮我开发一个股票交易机器人"
  ChatGPT: "好的，你需要什么功能？"
  用户: "嗯...我也不太清楚...大概需要情绪分析？"     ← 需要领域知识才能给出好 prompt
  ChatGPT: "好的，你想用什么编程语言？"
  用户: "Python 吧"
  ChatGPT: "好的，你想监控哪些数据源？"
  用户: "......"                                     ← 非专家很难回答
  
  问题：
  ✗ 用户需要持续参与，为每一步提供方向
  ✗ 非领域专家很难给出有效的引导
  ✗ 过程耗时且无法规模化
  ✗ 对话方向完全取决于人类 prompt 的质量
```

**CAMEL 的核心洞察：** 既然人类引导 LLM 的本质是"一个用户不断提问/指令，一个助手不断回答/执行"，那能不能**用另一个 LLM 来扮演用户**，从而把人类从循环中解放出来？

```
【传统模式 vs CAMEL 模式】

  传统:  人类用户 ←→ LLM 助手
         ↑ 人类需要持续在线

  CAMEL: 人类只需提供初始想法
           ↓
         AI 用户（LLM₁）←→ AI 助手（LLM₂）
         ↑ 两个 LLM 自主对话直到任务完成
         ↑ 人类可以离开了
```

## 这个问题真实存在吗？

**真实存在，且在 2023 年 3 月是 Agent 领域最迫切的问题之一。**

*   **Prompt 工程的高门槛：** 想让 LLM 开发一个交易机器人，你需要了解金融 API、情绪分析工具、Python 库生态等。不具备这些知识的人根本写不出有效的 prompt。
*   **人类在环的不可扩展性：** 如果每个任务都需要人类全程参与，那 LLM 本质上只是一个更智能的搜索引擎，无法成为自主的 Agent。要让 Agent 大规模部署，必须减少对人类引导的依赖。
*   **多 Agent 协作的理论价值：** Marvin Minsky 在《Society of Mind》中提出，智能不来自单一的完美原理，而来自"多样性的集合"。如果两个 Agent 能自主协作，我们就有了研究"LLM 社会"行为和认知过程的工具。

## 它是如何解决的？

### 1. 角色扮演框架：全局架构

CAMEL 的完整流程分三个阶段：

```
┌─────────────────────────────────────────────────────────────┐
│                   CAMEL 角色扮演框架                          │
│                                                             │
│  阶段 1: 人类输入（唯一需要人参与的环节）                     │
│  ┌────────────────────────────────────────────────┐         │
│  │  人类提供:                                      │         │
│  │  ① 一个初始想法（如"开发股票交易机器人"）         │         │
│  │  ② 两个角色（如 Python 程序员 + 股票交易员）     │         │
│  └────────────────────┬───────────────────────────┘         │
│                       │                                     │
│  阶段 2: 任务具体化（Task Specifier Agent）                  │
│  ┌────────────────────▼───────────────────────────┐         │
│  │  Task Specifier 将模糊想法变成具体任务:           │         │
│  │                                                 │         │
│  │  输入: "开发股票交易机器人"                       │         │
│  │  输出: "开发一个带有情绪分析工具的交易机器人，    │         │
│  │         能监控社交媒体上对特定股票的正面或          │         │
│  │         负面评论，并基于情绪分析结果执行交易"      │         │
│  └────────────────────┬───────────────────────────┘         │
│                       │                                     │
│  阶段 3: 自主协作（核心循环——人类不再参与）                   │
│  ┌────────────────────▼───────────────────────────┐         │
│  │                                                 │         │
│  │   AI 用户（股票交易员）                           │         │
│  │   ┌─────────────────┐                           │         │
│  │   │ 负责: 拆解任务    │                           │         │
│  │   │ 下达具体指令      │ ──→ Instruction          │         │
│  │   │ 判断任务是否完成  │                           │         │
│  │   └─────────────────┘                           │         │
│  │          ↑                        │              │         │
│  │     Solution                 Instruction         │         │
│  │          │                        ↓              │         │
│  │   ┌─────────────────┐                           │         │
│  │   │ 负责: 执行指令    │                           │         │
│  │   │ 提供具体方案      │ ──→ Solution             │         │
│  │   │ 请求下一步指令    │                           │         │
│  │   └─────────────────┘                           │         │
│  │   AI 助手（Python 程序员）                        │         │
│  │                                                 │         │
│  │   循环直到 AI 用户说 <CAMEL_TASK_DONE>            │         │
│  └─────────────────────────────────────────────────┘         │
└─────────────────────────────────────────────────────────────┘
```

这个设计的精妙之处在于**角色分工**：
*   **AI 用户** = 任务规划者（Task Planner）：它不执行任何操作，只负责"想好下一步该做什么"并下达指令
*   **AI 助手** = 任务执行者（Task Executor）：它不做规划，只负责"收到指令就执行，给出具体方案"

这种分工模拟了现实中"产品经理 + 工程师"的协作模式——一个人负责"做什么"，另一个负责"怎么做"。

### 2. Inception Prompting：让自主协作成为可能的核心技术

CAMEL 能工作的根本原因不是框架设计，而是**精心工程化的系统提示**。论文把这种技术称为"Inception Prompting"（盗梦提示）——灵感来自电影《盗梦空间》中"在人的脑中植入一个想法"的概念。

Inception Prompt 由三部分组成：

```
Inception Prompt = 任务指定提示 (P_T) + 助手系统提示 (P_A) + 用户系统提示 (P_U)
```

**关键设计：** 所有 prompt 工程只发生在对话开始前（系统消息阶段）。一旦对话开始，AI 助手和 AI 用户**互相提示**，形成自动循环——这就是"Inception"的含义：**初始的提示"种子"会自我传播，驱动整个对话流程**。

#### 助手系统提示的关键设计

论文详细解析了每条 prompt 规则的设计动机。每一条都是为了解决自主协作中发现的具体问题：

```
┌─────────────────────────────────────────────────────────────┐
│  助手系统提示 (P_A) 的关键规则及其解决的问题                   │
│                                                             │
│  规则 1: "Never forget you are a {助手角色} and              │
│          I am a {用户角色}. Never flip roles!"               │
│                                                             │
│  → 解决: 角色翻转（Role Flipping）                           │
│    实验中发现助手会突然开始下达指令、用户开始执行                │
│    两个 Agent 的角色互换导致对话混乱                           │
│                                                             │
│  规则 2: "Never instruct me!"                               │
│                                                             │
│  → 解决: 助手抢夺控制权                                      │
│    助手有时会反过来问用户问题或给用户布置任务                   │
│                                                             │
│  规则 3: "You must decline my instruction honestly if        │
│          you cannot perform the instruction due to           │
│          physical, moral, legal reasons..."                  │
│                                                             │
│  → 解决: 有害内容 + 对齐问题                                 │
│    防止 Agent 产生虚假、有害或违法的输出                       │
│                                                             │
│  规则 4: "Unless I say the task is completed, you should     │
│          always start with: Solution: <YOUR_SOLUTION>"       │
│                                                             │
│  → 解决: 模糊回复（Flake Replies）                           │
│    实验中助手经常回复"I will do something"这种                │
│    承诺但不行动的空话，强制格式迫使它给出具体方案              │
│                                                             │
│  规则 5: "Always end your solution with: Next request."      │
│                                                             │
│  → 解决: 对话停滞                                            │
│    确保助手主动请求下一条指令，维持对话循环                    │
└─────────────────────────────────────────────────────────────┘
```

#### 用户系统提示的关键设计

```
┌─────────────────────────────────────────────────────────────┐
│  用户系统提示 (P_U) 的关键规则                               │
│                                                             │
│  规则 1: "You must instruct me ... ONLY in the following    │
│          two ways:                                           │
│          1. Instruct with a necessary input: ...            │
│          2. Instruct without any input: ..."                │
│                                                             │
│  → 目的: 标准化指令格式                                      │
│    使生成的指令-方案对可以直接用于微调 LLM                    │
│    （Instruction + Input 的格式与 Alpaca 等数据集兼容）       │
│                                                             │
│  规则 2: "When the task is completed, you must only reply   │
│          with a single word <CAMEL_TASK_DONE>."             │
│                                                             │
│  → 解决: 无限消息循环（Infinite Loop）                       │
│    实验中发现两个 Agent 会陷入互相说"谢谢"/"再见"            │
│    的死循环——引入终止 token 让用户能主动结束对话              │
│    有趣的是，有时 Agent 意识到自己在循环中                    │
│    却无法自行跳出                                            │
└─────────────────────────────────────────────────────────────┘
```

### 3. 对话的形式化描述

CAMEL 的自主协作过程可以用简洁的数学语言描述：

$$M_t = \{(I_0, S_0), (I_1, S_1), ..., (I_t, S_t)\}$$

其中 $I_t$ 是时刻 $t$ 的用户指令，$S_t$ 是助手方案。每一轮的更新规则：

$$I_{t+1} = U(M_t) \quad\quad S_{t+1} = A(M_t, I_{t+1})$$

其中 $U$ 是用户 Agent（基于历史对话生成下一条指令），$A$ 是助手 Agent（基于历史对话和新指令生成方案）。对话历史通过累积更新：

$$M_{t+1} = M_t \cup \{(I_{t+1}, S_{t+1})\}$$

循环持续直到用户 Agent 输出 `<CAMEL_TASK_DONE>` 或触发其他终止条件。

```
一次完整对话的实际样例（开发交易机器人）：

  AI 用户 (股票交易员):
    Instruction: 安装情绪分析和股票交易所需的 Python 库。
    Input: None

  AI 助手 (Python 程序员):
    Solution: 我们需要安装以下库：
    1. tweepy — 访问 Twitter API
    2. textblob — 情绪分析
    3. yfinance — 访问 Yahoo Finance API
    
    pip install tweepy
    pip install textblob
    pip install yfinance
    
    Next request.

  AI 用户:
    Instruction: 在 Python 中导入这些库。
    Input: None

  AI 助手:
    Solution:
    ```python
    import tweepy
    from textblob import TextBlob
    import pandas as pd
    import numpy as np
    import yfinance as yf
    ```
    Next request.

  ... （持续多轮直到完成整个交易机器人的开发）

  AI 用户:
    <CAMEL_TASK_DONE>
```

### 4. 自主协作的四大挑战

论文在大规模实验中发现了四个反复出现的问题，这些发现对后续所有多 Agent 系统都有重要参考价值：

```
┌────────────────┬──────────────────────────────────────────────┐
│  挑战            │  现象描述                                     │
├────────────────┼──────────────────────────────────────────────┤
│  角色翻转        │  助手突然开始下达指令，用户开始执行方案          │
│  (Role Flipping) │  两个 Agent 的角色互换，对话走向混乱           │
├────────────────┼──────────────────────────────────────────────┤
│  复读机          │  助手不提供方案，只是重复用户的指令             │
│  (Repeating)    │  "好的，我会安装这些库" ← 但不给代码           │
├────────────────┼──────────────────────────────────────────────┤
│  模糊回复        │  助手承诺行动但不执行:                         │
│  (Flake Replies) │  "I will implement the function..." ← 空话  │
│                 │  没有任何具体代码或方案                        │
├────────────────┼──────────────────────────────────────────────┤
│  无限循环        │  两个 Agent 互相说"谢谢"/"再见"陷入死循环      │
│  (Infinite Loop) │  有时 Agent 知道自己在循环中但无法跳出         │
└────────────────┴──────────────────────────────────────────────┘

  → 这四个问题几乎是所有多 Agent 系统的"公敌"
  → CAMEL 通过 Inception Prompting 的精心设计逐一解决
  → 但根本原因在于 LLM 的指令遵循能力有限
    （更强的模型如 GPT-4 这些问题会大幅减少）
```

### 5. 终止条件设计

为了确保对话能正常结束，CAMEL 设计了五个终止条件：

```
五个终止条件（满足任一则结束对话）：

  ① 用户无指令: 用户连续 3 轮没有给出新指令
     → 说明用户 Agent "卡住了"，无法继续规划

  ② 助手反客为主: 助手开始给用户下指令
     → 检测到角色翻转，立即终止

  ③ 任务完成 token: 用户输出 <CAMEL_TASK_DONE>
     → 正常完成，用户判断任务已解决

  ④ Token 上限: 达到 gpt-3.5-turbo 的上下文长度限制
     → 硬性约束，无法继续

  ⑤ 消息数上限: 达到 40 条消息（成本控制）
     → 对话成本随长度二次增长，必须设上限
```

### 6. Critic-In-The-Loop：可选的质量控制

论文还提出了一个扩展机制——引入第三个"批评者"Agent，可以在对话过程中选择最佳提案或提供反馈，使决策过程类似于树搜索：

```
基础模式（双 Agent）：

  AI 用户 ──指令──→ AI 助手
          ←─方案──

扩展模式（加入 Critic）：

  AI 用户 ──指令──→ AI 助手 ──方案──→ Critic
          ←─方案──                    │
                                      ▼
                              "方案 B 更好" 或
                              "需要补充错误处理"
                                      │
                              反馈注入下一轮对话
```

Critic 可以是另一个 LLM，也可以是人类——这为框架提供了灵活的质量控制层。

## 实验结果

### 大规模数据生成

CAMEL 的一个重要贡献是证明了角色扮演可以**规模化地生成高质量对话数据**：

```
数据集生成规模：

  AI Society 数据集:
    50 个助手角色 × 50 个用户角色 × 10 个任务/组合
    = 25,000 段多轮对话

  Code 数据集:
    编程相关的助手-用户协作对话

  Math 数据集:
    数学问题-解答对

  Science 数据集:
    科学问题-解答对

  → 角色和任务本身也是由 LLM 自动生成的
  → 整个数据管线几乎不需要人类参与
```

### Agent 质量评估

论文对 CAMEL 的协作方案 vs 单次 gpt-3.5-turbo 方案进行了对比评估：

```
评估结果（100 个任务）：

                       平局     gpt-3.5 胜    CAMEL 胜
  ─────────────────────────────────────────────────────
  AI Society (人类评估)  13.3%     10.4%        76.3%
  AI Society (GPT-4评估)  4.0%     23.0%        73.0%
  Code (GPT-4 评估)       0.0%     24.0%        76.0%

  关键发现:
  1. CAMEL 协作方案以约 3:1 的比例碾压单次生成
  2. 人类评估和 GPT-4 评估高度一致
  3. 多 Agent 协作 > 单 Agent 单次生成（即使用同一个模型）
```

这个结果的意义深远：**同样的 gpt-3.5-turbo，两个实例通过角色扮演协作，产出质量远超单个实例的一次性生成。** 这说明"协作框架"本身就能显著提升 LLM 的有效输出质量。

### 知识涌现实验

论文还做了一个有趣的实验——用 CAMEL 生成的数据集渐进式微调 LLaMA-7B，观察知识的"涌现"：

```
渐进式微调路径：

  LLaMA-7B → +AI Society → +Code → +Math → +Science
                  ↓            ↓        ↓         ↓
             学会社交对话   学会写代码  学会算数  学会科学知识

  有趣发现:
  • 加入 Code 数据后，Science 能力也提升了
    （因为 Code 数据中包含科学领域的编程任务）
  • 加入 AI Society 后，Code 能力也提升了
    （因为社交数据中包含"程序员"角色的对话）
  → 不同领域的数据之间存在正向迁移
```

最终的 CAMEL-7B（LLaMA-7B 微调版）在 HumanEval 上达到 14.0% pass@1，超越了原始 LLaMA-7B（10.5%）和 Vicuna-7B（11.0%）。

## 深层洞察

### 1. "让 Agent 互相提示" 的范式意义

CAMEL 最重要的贡献不是具体的技术实现，而是提出了一个全新的范式——**Agent-to-Agent Prompting**。在 CAMEL 之前，prompt 都是人类写给 LLM 的；CAMEL 证明了 LLM 可以互相提示，形成自治的协作系统。这个思路直接影响了后续大量工作：

```
CAMEL 开创的范式对后续工作的影响：

  CAMEL (2023.03): 两个 Agent 角色扮演，互相提示
     ↓
  Generative Agents (2023.04): 25 个 Agent 在虚拟小镇中自主互动
     ↓
  MetaGPT (2023.08): 多个 Agent 模拟软件公司各角色协作
     ↓
  AutoGen (2023.10): 微软的多 Agent 对话框架
     ↓
  CrewAI (2024): 角色驱动的多 Agent 协作框架

  → CAMEL 是这条"多 Agent 协作"技术路线的起点之一
```

### 2. Prompt 工程的"冰山效应"

CAMEL 的论文看起来"简单"——不就是两个 LLM 互相聊天吗？但实际上，**论文中最核心的贡献是那些 prompt 规则**，而这些规则的背后是大量的失败实验。每条"Never flip roles!"这样看似简单的规则，都是在观察到具体的失败模式后反复迭代得出的。

这揭示了一个深刻的事实：**在 LLM Agent 的世界里，80% 的工作量在 prompt 工程上，而不在架构设计上。** 框架图画出来都很简洁优雅，但让它实际跑起来的关键是那些"脏活"——处理角色翻转、防止无限循环、强制输出格式等。

### 3. "AI 社会" 研究的先声

CAMEL 论文的副标题"Mind Exploration of Large Language Model Society"暗示了一个更宏大的愿景——把多 Agent 系统当作一个"社会"来研究。论文中观察到的现象（角色翻转、无限循环、Agent 知道自己在循环中却无法跳出）本身就是关于 LLM "认知"过程的有趣发现。

论文还专门生成了一个 **Misalignment 数据集**，模拟了不对齐的自主 Agent 系统可能带来的潜在风险——这在 2023 年 3 月是非常有前瞻性的安全研究。

## CAMEL 的局限性

*   **对话质量受底座模型限制：** 使用 gpt-3.5-turbo 时四大挑战频繁出现，更强的模型（如 GPT-4）会大幅缓解。
*   **只有双 Agent 场景：** 论文只研究了 1 对 1 的助手-用户协作，没有扩展到多角色、多团队的复杂协作场景。
*   **缺乏外部工具交互：** Agent 只在对话中生成文本方案，没有实际执行代码、调用 API 或与真实环境交互的能力。
*   **成本随对话长度二次增长：** 每一轮都需要将完整历史传入 LLM，长对话的成本快速膨胀。

---

## 总结

CAMEL 的核心贡献可以概括为三个层面：

1. **范式层面：** 提出了"角色扮演 + Inception Prompting"的自主多 Agent 协作框架，证明了两个 LLM 可以在几乎不需要人类介入的情况下自主完成复杂任务。
2. **工程层面：** 系统性地识别并解决了自主协作中的四大挑战（角色翻转、复读、模糊回复、无限循环），为后续所有多 Agent 系统提供了实践参考。
3. **数据层面：** 证明了角色扮演可以规模化生成高质量对话数据（25,000+ 段），并通过渐进式微调展示了知识涌现。

CAMEL 是 2023 年多 Agent 协作浪潮的起点之一。它回答了一个看似简单实则深刻的问题：**当两个 LLM 被赋予互补的角色并开始对话时，它们能自主协作完成人类都觉得复杂的任务——而这一切的秘密，藏在精心设计的初始提示中。**
