*论文发布时间：2022-10-06*

Princeton 和 Google Brain 联合发布的 **ReAct**（论文标题为《[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)》）是 AI Agent 领域的奠基性工作之一。
简单来说，这篇论文提出了一种让大语言模型（LLM）**同时生成推理轨迹（Reasoning Traces）和任务动作（Actions）** 的范式，让"思考"和"行动"交替进行、相互增强——推理帮助模型制定和调整行动计划，行动则让模型与外部环境交互获取新信息来修正推理。

## ReAct 解决了什么问题？

![Figure 1: ReAct 范式对比。(1a) Reason Only——模型只推理不行动，容易产生幻觉；(1b) Act Only——模型只行动不推理，缺乏策略；(1c) ReAct——推理与行动交替进行，内部推理引导外部行动，外部观察校准内部推理；(1d) ReAct 内部循环细节。](https://react-lm.github.io/files/diagram.png)

**核心问题：LLM 的"推理"和"行动"是割裂的。**

在 ReAct 之前，LLM 的两大能力方向各自独立发展，形成了两个互不相通的阵营：

*   **只推理（Reason-only）：** 以 Chain-of-Thought（CoT）为代表。模型通过自我对话生成推理链来回答问题，但**完全闭门造车**——它只能利用预训练时记住的知识，无法访问外部世界来验证或获取新信息。结果就是容易产生**幻觉（Hallucination）**和**错误累积（Error Propagation）**。
*   **只行动（Act-only）：** 以 WebGPT、SayCan 为代表。模型直接将文本上下文映射为具体动作（如搜索、点击），但**缺乏抽象推理能力**——它不会在行动前思考"为什么要这么做"，也不会在遇到问题时反思调整策略。

```
【两个阵营的困境】

阵营 1: 只推理（CoT）                    阵营 2: 只行动（Act-only）
┌─────────────────────────┐          ┌──────────────────────────┐
│  问: Apple Remote 还能    │          │  问: Apple Remote 还能     │
│  控制什么设备？            │          │  控制什么设备？             │
│                          │          │                           │
│  想法1: Apple Remote 是    │          │  动作1: Search[Apple Remote]│
│  苹果设计的遥控器...      │          │  观察1: Apple Remote 是...  │
│  想法2: 它能控制 Mac 和    │          │  动作2: Search[Front Row]   │
│  Apple TV... ← 猜测!      │          │  观察2: Front Row 是...     │
│  想法3: 所以答案是         │          │  动作3: Finish[???]         │
│  Mac 和 Apple TV  ← 幻觉! │          │  ← 有了信息但不会综合推理!   │
│                          │          │                           │
│  ✗ 不查外部资料，纯靠猜    │          │  ✗ 不思考为什么，只机械执行   │
└─────────────────────────┘          └──────────────────────────┘
```

**这和人类解决问题的方式完全不同。** 人类天生具有将推理与行动无缝交织的能力——你在超市购物时，会一边思考"还缺什么菜"（推理），一边拿起食材查看保质期（行动），然后根据观察到的信息调整采购计划（推理更新）。ReAct 要做的，就是让 LLM 也学会这种"边想边做"的能力。

## 这个问题真实存在吗？

**真实存在，而且在 2022 年是制约 LLM 实用化的核心瓶颈。**

*   **CoT 的幻觉问题严重：** 在 HotpotQA（多跳问答）任务上，CoT 产生的推理看起来流畅合理，但仔细检查会发现大量事实错误。模型会编造看起来"像那么回事"的推理链，却与现实完全不符。
*   **Action-only 的推理缺失：** 在 ALFWorld（文本冒险游戏）中，纯动作模型无法将高层目标分解为子目标。当它需要"把杯子放到桌上"时，它不会先思考"杯子在哪里"，只是盲目搜索，效率极低。
*   **实际应用需要两者融合：** 无论是客服机器人（需要查知识库+推理回答）、代码助手（需要查文档+规划实现），还是自动化运维（需要读日志+分析定位），都需要"推理"和"行动"的紧密配合。

## 它是如何解决的？

ReAct 的核心思想极为简洁：**让模型在一个统一的序列中交替生成"想法（Thought）"和"动作（Action）"，并接收来自环境的"观察（Observation）"。**

### 1. Thought-Action-Observation 循环

这是 ReAct 的核心机制。模型在一个循环中交替进行三种操作：

```
┌────────────────────────────────────────────────────────────┐
│                  ReAct 核心循环                              │
│                                                            │
│   ┌─────────┐     ┌─────────┐     ┌──────────┐            │
│   │ Thought │────→│ Action  │────→│Observation│───┐        │
│   │ (推理)   │     │ (行动)  │     │ (观察)    │   │        │
│   └────▲────┘     └─────────┘     └──────────┘   │        │
│        │                                          │        │
│        └──────────────────────────────────────────┘        │
│                                                            │
│  Thought: 模型的内部推理（不影响外部环境）                    │
│           - 分解目标、制定计划                               │
│           - 从观察中提取关键信息                              │
│           - 跟踪进度、处理异常                               │
│                                                            │
│  Action:  模型对外部环境的操作（产生实际效果）                 │
│           - Search[query]: 搜索维基百科                      │
│           - Lookup[keyword]: 在当前页面查找                   │
│           - Finish[answer]: 提交最终答案                     │
│                                                            │
│  Observation: 环境返回的反馈（模型无法控制）                   │
│           - 搜索结果、页面内容、操作反馈等                     │
└────────────────────────────────────────────────────────────┘
```

### 2. 一个具体的例子

![Figure 2: HotpotQA 多跳问答中 ReAct 的实际轨迹对比。展示了 Standard、CoT（只推理）、Act（只行动）和 ReAct（推理+行动交替）四种方法在同一问题上的不同表现。ReAct 通过 Thought 引导搜索方向，通过 Observation 获取真实信息，避免了幻觉。](https://react-lm.github.io/files/hotpotqa.png)

以 HotpotQA 的多跳问答为例，看看 ReAct 如何一步步解决问题：

```
问题: Colorado orogeny 的东部延伸到了哪个区域？该区域的海拔范围是多少？

━━ 标准 CoT（只推理）━━

  Thought 1: Colorado orogeny 是发生在科罗拉多的造山运动...
  Thought 2: 它的东部可能延伸到大平原地区...  ← 猜测
  Thought 3: 大平原的海拔大约是 2000-5000 英尺... ← 猜测
  Answer: 2,000 to 5,000 ft  ← 可能是错的

━━ ReAct（推理 + 行动）━━

  Thought 1: 我需要搜索 Colorado orogeny，找到东部延伸的区域，
             然后查该区域的海拔范围。
  Action 1:  Search[Colorado orogeny]
  Obs 1:     Colorado orogeny 是发生在科罗拉多及周边地区的造山运动...

  Thought 2: 搜索结果没提到东部。我需要查找"eastern sector"。
  Action 2:  Lookup[eastern sector]
  Obs 2:     东部延伸到 High Plains，被称为 Central Plains orogeny。

  Thought 3: 东部延伸到 High Plains。现在我需要搜索 High Plains
             的海拔范围。
  Action 3:  Search[High Plains]
  Obs 3:     High Plains 指两个不同的地区之一...

  Thought 4: 搜索结果不够具体，我需要搜索 High Plains (United States)。
  Action 4:  Search[High Plains (United States)]
  Obs 4:     High Plains 从东到西海拔从约 1,800 升至 7,000 英尺。

  Thought 5: 找到了！海拔范围是 1,800 到 7,000 英尺。
  Action 5:  Finish[1,800 to 7,000 ft]  ← 有据可查的正确答案
```

对比一下这两种方式的关键差异：

```
┌──────────────┬────────────────────────┬──────────────────────────┐
│              │  CoT（只推理）          │  ReAct（推理+行动）        │
├──────────────┼────────────────────────┼──────────────────────────┤
│  知识来源     │  仅限模型内部记忆        │  内部记忆 + 外部检索       │
│  事实准确性   │  容易幻觉              │  有外部信息做支撑（Grounded）│
│  错误修正     │  错误会沿推理链累积      │  每步都有观察来纠偏         │
│  可解释性     │  推理过程不透明          │  Thought 清晰记录决策理由   │
│  灵活性       │  推理格式自由但不接地    │  格式有结构约束但可靠       │
└──────────────┴────────────────────────┴──────────────────────────┘
```

### 3. 两种任务模式

ReAct 根据任务性质采用不同的推理密度：

```
【模式 A: 知识密集型任务（HotpotQA, FEVER）—— 推理与行动密集交替】

  Thought 1 → Action 1 → Obs 1 → Thought 2 → Action 2 → Obs 2 → ...
  │            │                   │            │
  每步都思考    每步都行动            每步都思考    每步都行动
  (推理密集)

  适用场景: 需要多步检索+综合推理的问答和事实验证任务
  推理作用: 分解问题、提取信息、引导下一步搜索方向、综合最终答案


【模式 B: 交互决策型任务（ALFWorld, WebShop）—— 推理稀疏出现】

  Act 1 → Obs 1 → Act 2 → Obs 2 → ... → Thought → Act N → ...
  │                │                       │
  直接行动          直接行动                 偶尔思考
  (行动密集)

  适用场景: 需要大量环境交互的游戏和导航任务
  推理作用: 在关键节点进行反思和策略调整
  由模型自主决定何时插入推理
```

### 4. ReAct 的 Prompt 设计

ReAct 的实现完全基于 **Few-shot Prompting**（少样本提示），不需要对模型进行任何微调。核心做法是：

```
━━ Prompt 结构 ━━

  ┌──────────────────────────────────────────────────┐
  │  系统指令（隐式，通过示例传达格式）                  │
  ├──────────────────────────────────────────────────┤
  │  示例 1: 完整的 Thought-Action-Obs 轨迹            │
  │  示例 2: 完整的 Thought-Action-Obs 轨迹            │
  │  ...                                             │
  │  示例 N: 完整的 Thought-Action-Obs 轨迹            │
  ├──────────────────────────────────────────────────┤
  │  新问题: <用户的实际问题>                           │
  │  Thought 1:  ← 模型从这里开始生成                  │
  └──────────────────────────────────────────────────┘

  关键设计点:
  - 示例中的 Thought 和 Action 是人工撰写的
  - Observation 是真实的环境返回结果
  - HotpotQA 用 6 个示例，FEVER 用 3 个
  - ALFWorld 用 2 个，WebShop 用 1 个
  - 模型通过模仿示例的格式，自动学会交替生成
```

**Thought 的六种功能**（论文中归纳的推理轨迹类型）：

```
┌─────────────────────────────────────────────────────────────┐
│  1. 目标分解     "我需要先找 X，再查 Y，最后比较 Z"           │
│  2. 常识注入     "因为 X 和 Y 是同义词，所以我应该搜索..."     │
│  3. 信息提取     "从观察结果中，关键信息是..."                 │
│  4. 进度追踪     "我已经找到了 A，还需要找 B"                  │
│  5. 搜索引导     "上次搜索结果不理想，换个关键词试试"           │
│  6. 答案综合     "综合以上信息，最终答案是..."                 │
└─────────────────────────────────────────────────────────────┘
```

## 实验结果

### 知识密集型任务

在 HotpotQA 和 FEVER 上，使用 PaLM-540B 作为基座模型：

```
┌────────────────────────┬────────────────┬────────────────┐
│  方法                   │ HotpotQA (EM)  │ FEVER (Acc)    │
├────────────────────────┼────────────────┼────────────────┤
│  Standard (直接回答)     │  28.7          │  57.1          │
│  CoT (只推理)           │  29.4          │  56.3          │
│  Act-only (只行动)       │  25.7          │  58.9          │
│  ReAct                  │  27.4          │  60.9          │
│  ReAct + CoT-SC (最佳)  │  35.1          │  64.6          │
├────────────────────────┼────────────────┼────────────────┤
│  监督学习 SOTA          │  67.5 (~140k)  │  89.5 (~90k)   │
└────────────────────────┴────────────────┴────────────────┘

  注: CoT-SC = Chain-of-Thought + Self-Consistency
  ReAct + CoT-SC 是论文发现的最佳组合策略
```

值得注意的是，ReAct 在 HotpotQA 上的数值略低于 CoT，但在 FEVER 上显著超越。论文对此做了详细的错误分析：

```
ReAct 在 HotpotQA 上的两类失败模式:

  1. 检索噪声 (搜索结果不相关)
     → 搜索引擎返回了无用信息
     → 模型被噪声干扰，推理脱轨
     → 难以恢复和重新规划搜索策略

  2. 结构约束 (Thought-Action 格式的限制)
     → 交替生成的格式一定程度上限制了推理的自由度
     → 有些复杂推理需要连续多步 Thought 才能完成

CoT 的典型失败模式:

  1. 事实幻觉 (编造不存在的事实)
     → 这恰好是 ReAct 通过外部检索能避免的
     → 所以两者互补: ReAct + CoT-SC 效果最佳
```

### 交互决策型任务

![Figure: ALFWorld 文本冒险游戏中 Act-only 与 ReAct 的对比。Act-only 盲目搜索每个位置；ReAct 通过 Thought 进行目标分解和常识推理（"桌灯通常在书桌或梳妆台上"），大幅减少无效行动。](https://react-lm.github.io/files/alfworld.png)

在 ALFWorld 和 WebShop 上的表现：

```
┌────────────────────────┬────────────────┬────────────────┐
│  方法                   │ ALFWorld (SR)  │ WebShop (SR)   │
├────────────────────────┼────────────────┼────────────────┤
│  Act-only (PaLM)        │  45%           │  30.1%         │
│  ReAct (PaLM, 2-shot)   │  71%           │  40.0%         │
├────────────────────────┼────────────────┼────────────────┤
│  模仿学习 Baseline      │  37% (~100k)   │  29.1% (~90k)  │
│  强化学习 Baseline      │  —             │  28.7%         │
└────────────────────────┴────────────────┴────────────────┘

  SR = Success Rate (成功率)
  ReAct 仅用 1-2 个示例，就超越了用 ~10 万条数据训练的方法
```

这个结果非常惊人：**仅用 2 个示例做 Prompt 的 ReAct，在 ALFWorld 上比用 10 万条样本训练的模仿学习高出 34 个百分点。** 这说明推理能力对交互决策的帮助是巨大的——模型不再是盲目尝试，而是"先想后做"。

![Figure: HotpotQA 和 FEVER 上各方法的性能对比。ReAct 在 FEVER 上显著超越 CoT，两者的最佳组合（ReAct → CoT-SC 切换策略）在所有设置中取得最优。](https://react-lm.github.io/files/results.png)

### Fine-tuning 结果

![Figure: 微调实验结果。使用 ReAct 轨迹微调的小模型（PaLM-8/62B）在多个指标上超越了使用 Prompt 的大模型（PaLM-540B），证明 ReAct 格式是最佳的微调数据格式。](https://react-lm.github.io/files/hotpot_finetune.png)

论文还探索了用 ReAct 格式的轨迹来微调较小模型：

```
训练数据生成流程:

  PaLM-540B (ReAct Prompt)
       │
       ├── 生成大量 Thought-Action-Obs 轨迹
       │
       ├── 筛选成功完成任务的轨迹
       │
       └── 用这些轨迹微调 PaLM-8B / PaLM-62B

关键发现:
  1. ReAct 是所有模型尺寸下最佳的微调格式
  2. 用 ReAct 微调的小模型 > 用 Prompt 的大模型
     (微调的 PaLM-8B 在某些指标上超过 Prompt 的 PaLM-540B)
```

## 为什么 ReAct 如此重要？

### 1. 奠定了 AI Agent 的基本范式

ReAct 首次证明了"推理+行动"的交替范式在多种任务上的通用性。这个思想直接影响了后来几乎所有 AI Agent 框架的设计：

```
ReAct (2022) 的影响脉络:

  ReAct
    │
    ├── LangChain (2022.10)  ← 将 ReAct 封装为标准 Agent 接口
    │
    ├── Reflexion (2023)     ← 在 ReAct 基础上增加自我反思
    │
    ├── Toolformer (2023)    ← 让模型学会自主调用工具
    │
    ├── AutoGPT (2023)       ← 全自动 Agent，核心循环借鉴 ReAct
    │
    └── Manus (2025)         ← 多 Agent 协作，每个 Agent 内部仍是
                                Thought-Action-Observation 循环
```

### 2. 人类可解释性和可控性

ReAct 的 Thought 轨迹是**完全透明的**。人类可以：

*   **检查**：阅读每一步 Thought，理解模型为什么这么做
*   **诊断**：定位出错的具体步骤（是推理错了还是搜索错了）
*   **修正**：直接编辑某一步 Thought，模型会自动调整后续行为

论文展示了一个 Human-in-the-Loop（人在回路）的案例：在 ALFWorld 中，模型因为一个幻觉 Thought 而卡住。人类只需修改两句 Thought，模型就自动恢复正常并完成任务。**这种"修改想法就能纠正行为"的能力，是纯 Action 模型不具备的。**

### 3. Grounding：连接内部知识与外部世界

ReAct 实现了 LLM 的**接地（Grounding）**——模型的推理不再是闭门造车，而是持续从真实世界获取反馈来校准自己：

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│     LLM 内部知识                 外部世界                │
│  ┌──────────────┐          ┌──────────────────┐         │
│  │ 预训练记忆    │          │  Wikipedia API   │         │
│  │ 常识推理能力  │◄────────►│  搜索引擎        │         │
│  │ 语言理解     │  ReAct    │  数据库          │         │
│  │ 逻辑推演     │  桥梁     │  文件系统        │         │
│  └──────────────┘          │  浏览器          │         │
│                            │  代码执行环境     │         │
│                            └──────────────────┘         │
│                                                         │
│  CoT: 只用左边  →  容易幻觉                              │
│  Act: 只用右边  →  缺乏推理                              │
│  ReAct: 两边互通 →  推理有据，行动有谋                     │
└─────────────────────────────────────────────────────────┘
```

## ReAct 的局限性

论文也诚实地指出了 ReAct 的不足：

*   **对检索质量敏感：** 如果搜索引擎返回的信息不相关或有误导性，模型的推理会受到干扰。CoT 在这种情况下反而可能更好（因为它完全依赖内部知识）。
*   **推理格式的结构约束：** Thought-Action 的交替格式限制了推理的灵活性。某些需要长链推理的问题，可能需要连续多步 Thought。
*   **Prompt 长度限制：** Few-shot 示例会占用大量 context window，限制了可用于推理的空间。
*   **最佳策略是混合的：** 论文发现 ReAct + CoT-SC 的组合效果最好，说明单靠 ReAct 还不够，需要与其他方法互补。

---

### 总结

ReAct 的核心贡献可以用一句话概括：**让 LLM 学会"边想边做"——在推理中行动，在行动中推理。**

1. **范式创新**：首次将推理和行动统一在一个交替生成的序列中，证明了两者的协同效应。
2. **Grounding**：通过与外部环境交互，解决了纯推理的幻觉问题。
3. **通用性**：同一个框架在知识问答、事实验证、文本游戏、网页购物四种截然不同的任务上都有效。
4. **可解释性**：Thought 轨迹让 AI 的决策过程对人类透明可控。
5. **高效性**：仅用 1-2 个示例就超越了用大量数据训练的模型。

ReAct 不只是一篇论文，它定义了 AI Agent 的基本范式——**Thought-Action-Observation 循环**，成为后来 LangChain、AutoGPT、Manus 等所有 Agent 框架的思想源头。
