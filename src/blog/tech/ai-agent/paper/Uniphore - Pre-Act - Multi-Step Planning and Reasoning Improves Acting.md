*论文发布时间：2025-05-19*

Uniphore 发布的 **Pre-Act**（论文标题为《[Pre-Act: Multi-Step Planning and Reasoning Improves Acting in LLM Agents](https://arxiv.org/abs/2505.09970)》）是对经典 ReAct 范式的重要升级。
简单来说，这篇论文指出 ReAct 的"一步一想"模式在复杂多步任务中力不从心，提出了一种**"先规划、再执行、边执行边修正"**的增强方案——让 Agent 在每一步行动前都生成一个涵盖所有后续步骤的**完整多步计划（Multi-Step Plan）**，并随着每步执行结果动态调整计划，直到任务完成。

## Pre-Act 解决了什么问题？

![Figure 1: ReAct 与 Pre-Act 的对比。ReAct 每步只生成单条推理和动作；Pre-Act 在每步都生成覆盖所有后续步骤的完整多步计划，并随执行结果动态修正。](https://arxiv.org/html/2505.09970v1/x1.png)

**核心问题：ReAct 的推理是"近视"的——只看眼前一步。**

ReAct 确立了 Thought-Action-Observation 循环的经典范式，但它的 Thought（推理）通常只关注**当前这一步应该做什么**，缺乏对整体任务的全局规划能力：

```
【ReAct 的"近视"问题】

  用户请求: "帮我查一下订单 #1234 的状态，如果有问题就申请退款"

  ━━ ReAct 的实际执行 ━━

  Thought 1: 我需要查询订单状态。           ← 只想了眼前一步
  Action 1:  query_order(order_id="1234")
  Obs 1:     订单状态: 已发货，物流异常

  Thought 2: 物流异常，我应该...嗯...       ← 每次从零开始思考下一步
  Action 2:  get_logistics_detail(order_id="1234")
  Obs 2:     快递已滞留 7 天

  Thought 3: 滞留 7 天，可能需要退款...      ← 仍然没有全局计划
  Action 3:  check_refund_eligibility(order_id="1234")
  ...

  问题:
  ✗ 没有预见到完整的任务链条
  ✗ 每步推理只考虑即时动作，不规划后续步骤
  ✗ 遇到意外情况（如某步失败）缺乏应变策略
  ✗ 对复杂的多步任务，经常迷失方向或遗漏步骤
```

这在简单任务中没有太大问题——查一个问题、搜一条信息，一两步就搞定了。但在**真实的企业级场景**中（客服工单处理、医疗问诊流程、金融合规审查），一个任务可能需要 5-10 步甚至更多的工具调用，步骤之间有复杂的依赖和分支逻辑。ReAct 的"走一步看一步"方式，在这种场景下效率低下且容易出错。

## 这个问题真实存在吗？

**真实存在，而且是 Agent 从实验室走向生产环境的核心瓶颈。**

*   **商业对话系统的需求：** 企业级 Agent 需要处理多轮对话、多工具协同、异常恢复等复杂场景。例如银行客服 Agent 处理一笔争议交易，可能需要：查交易记录 → 核实身份 → 检查争议政策 → 发起退款 → 确认结果。每步都依赖前一步的输出，且任何一步失败都需要调整后续计划。
*   **小模型的推理不足：** 先进的推理能力主要集中在 GPT-4、Claude 等私有大模型中。但企业部署中，出于**延迟和成本**考虑，往往需要使用 Llama 8B/70B 这样的小模型。这些模型在 ReAct 模式下，推理能力更加不足。
*   **论文的实验数据佐证：** 在 Almita 数据集（跨领域 Agent 测试集）上，ReAct 模式下五个模型的平均 Action Recall 仅为 **0.28**，而 Pre-Act 提升到 **0.47**——提升了 70%。

## 它是如何解决的？

### 1. 核心思想：从"一步一想"到"先规划再执行"

Pre-Act 的核心改变只有一个：**在每一步行动前，不仅思考当前步骤，还生成包含所有后续步骤的完整计划。**

```
━━ ReAct vs Pre-Act 对比 ━━

【ReAct: 每步只想当前】

  Step 1:
    Thought: 我需要查询订单。
    Action: query_order(#1234)
    Obs: 物流异常

  Step 2:
    Thought: 物流异常，查详情。        ← 只想了这一步
    Action: get_logistics(#1234)
    Obs: 滞留 7 天

  Step 3:
    Thought: 需要退款。               ← 又只想了这一步
    Action: apply_refund(#1234)
    ...


【Pre-Act: 每步都有完整计划】

  Step 1:
    Plan:                              ← 生成完整的多步计划
      Step 1: 查询订单状态 [当前]
      Step 2: 如果有问题，查物流详情
      Step 3: 检查退款资格
      Step 4: 发起退款申请
      Step 5: 生成最终回复
    Reasoning: 用户要求查订单并处理问题，
               需要完整的查询-诊断-处理流程。
    Action: query_order(#1234)
    Obs: 物流异常

  Step 2:
    Plan (更新版):                      ← 根据上一步结果修正计划
      Step 1: ✅ 查询订单状态 → 物流异常
      Step 2: 查物流详情 [当前]
      Step 3: 检查退款资格（因为物流异常）
      Step 4: 发起退款
      Step 5: 生成回复
    Reasoning: 订单确实有物流异常，需要进一步
               了解详情来决定后续处理方式。
    Action: get_logistics(#1234)
    Obs: 滞留 7 天

  Step 3:
    Plan (再次更新):                    ← 持续修正
      Step 1: ✅ 查询订单 → 物流异常
      Step 2: ✅ 查物流 → 滞留 7 天
      Step 3: 检查退款资格 [当前]
      Step 4: 发起退款（如果符合条件）
      Step 5: 生成回复
    Reasoning: 滞留 7 天已超出正常时限，
               很可能符合退款条件。
    Action: check_refund_eligibility(#1234)
    ...
```

关键区别一目了然：**Pre-Act 的每一步都有"上帝视角"——知道自己从哪来、在哪里、要到哪去。**

### 2. 增量式多步规划（Incremental Multi-Step Planning）

Pre-Act 的规划不是一次性的"死计划"，而是**每执行一步就根据实际结果动态修正**的"活计划"：

```
┌────────────────────────────────────────────────────────────────┐
│               Pre-Act 增量式规划流程                             │
│                                                                │
│  用户输入 + 工具定义 + 累积上下文 Ct                              │
│         │                                                      │
│         ▼                                                      │
│  ┌─────────────────────────────────┐                           │
│  │  LLM 生成完整计划 + 推理         │                           │
│  │                                 │                           │
│  │  Plan:                          │                           │
│  │    s₁: [思考+动作] ← 当前步骤    │                           │
│  │    s₂: [思考+动作]              │                           │
│  │    ...                          │                           │
│  │    sₙ: [思考+动作]              │                           │
│  │    s_fa: 最终回复                │                           │
│  └──────────┬──────────────────────┘                           │
│             │                                                  │
│             ▼                                                  │
│  ┌──────────────────┐     ┌────────────────┐                   │
│  │ 执行 s₁ 的 Action │────→│ 环境返回 Obs o₁ │                   │
│  └──────────────────┘     └───────┬────────┘                   │
│                                   │                            │
│             ┌─────────────────────┘                            │
│             │                                                  │
│             ▼  将 (a₁, o₁) 加入累积上下文 Ct                     │
│                                                                │
│  ┌─────────────────────────────────┐                           │
│  │  LLM 根据 Ct 重新生成计划        │  ← 关键：每步都重新规划    │
│  │                                 │                           │
│  │  Plan (修正版):                  │                           │
│  │    s₁: ✅ 已完成 (结果: o₁)     │                           │
│  │    s₂: [思考+动作] ← 当前步骤   │  ← 可能因 o₁ 而调整       │
│  │    s₃: [可能新增/删除/修改]     │                           │
│  │    ...                          │                           │
│  │    s_fa: 最终回复                │                           │
│  └──────────┬──────────────────────┘                           │
│             │                                                  │
│             ▼                                                  │
│  ┌──────────────────┐     ┌────────────────┐                   │
│  │ 执行 s₂ 的 Action │────→│ 环境返回 Obs o₂ │                   │
│  └──────────────────┘     └───────┬────────┘                   │
│                                   │                            │
│             ... 循环直到 s_fa ...                               │
│                                                                │
│             ▼                                                  │
│  ┌──────────────────┐                                          │
│  │ 输出最终回复 s_fa  │                                          │
│  └──────────────────┘                                          │
└────────────────────────────────────────────────────────────────┘
```

形式化描述：对于用户输入，Pre-Act 生成 n+1 步的计划 S = {s₁, s₂, ..., sₙ, s_fa}，其中：
- 每个步骤 sᵢ 包含**详细的推理（Thought）和动作（Action）**
- 当 sᵢ 涉及工具调用时，执行动作 aᵢ 并获得观察 oᵢ
- 下一步 sᵢ₊₁ 的生成会利用累积上下文 Ct = {(a₁, o₁), ..., (aᵢ, oᵢ)} 来**修正计划**
- 最后一步 s_fa 是最终回复

**这个"增量修正"机制至关重要：** 当某一步的实际结果偏离预期（比如工具调用失败、返回意外数据），Agent 可以在下一步的计划中动态调整策略，而不是盲目继续执行已经过时的计划。

### 3. 课程学习（Curriculum Learning）微调策略

Pre-Act 不仅是一个 Prompt 方法，论文还提出了一种**两阶段渐进式微调**策略，让小模型也能具备多步规划能力：

```
┌────────────────────────────────────────────────────────────┐
│                课程学习微调流程                               │
│                                                            │
│  阶段 1: 基础能力训练（ReAct 格式）                          │
│  ┌──────────────────────────────────────────────────┐      │
│  │  数据集: Glaive Function-Calling v2 (34 万条)      │      │
│  │  格式:   ReAct（简单的 Thought → Action）          │      │
│  │  目标:   让模型学会区分"何时调用工具"和"何时回复"    │      │
│  │  训练:   全量 LoRA 微调                            │      │
│  └─────────────────────┬────────────────────────────┘      │
│                        │                                   │
│                        ▼                                   │
│  阶段 2: 高级规划训练（Pre-Act 格式）                        │
│  ┌──────────────────────────────────────────────────┐      │
│  │  数据集: 自有标注数据集 (1852 条，100+ 领域)        │      │
│  │  格式:   Pre-Act（多步计划 + 详细推理）             │      │
│  │  标注:   每步的推理由专家人工撰写                    │      │
│  │  训练:   LoRA 增量微调（保留阶段 1 的知识）          │      │
│  └──────────────────────────────────────────────────┘      │
│                                                            │
│  为什么分两阶段？                                           │
│  ─────────────────                                         │
│  1. Glaive 数据量大(34万)但缺乏多步推理标注                  │
│     → 适合教"基本功"（工具调用格式、参数填充）               │
│  2. 自有数据量小(1852)但质量高、有完整推理标注               │
│     → 适合教"高级能力"（多步规划、动态调整）                 │
│  3. LoRA 只修改少量参数 → 阶段 2 不会覆盖阶段 1 的知识       │
│     (实验证明遗忘率 < 1%)                                    │
└────────────────────────────────────────────────────────────┘
```

这个策略的精髓在于：**先学走路（基本工具调用），再学跑步（多步规划推理）。** 数据标注成本也被大幅降低——只有 1852 条高质量数据需要人工标注推理过程。

### 4. 两级评估框架

```
│  Level 2: 端到端评估（End-to-End）                          │

│                                                            │
│  指标:                                                     │
│  ┌──────────────────┬─────────────────────────────┐        │

│                                                            │

│  Level 2 → 像考试的"整张卷子能不能及格"                     │
└────────────────────────────────────────────────────────────┘
```

## 深层洞察

### 1. 从 ReAct 到 Pre-Act：Agent 架构的演进逻辑

Pre-Act 不是对 ReAct 的否定，而是自然演进。可以用人类认知来类比：

```
┌────────────────────────────────────────────────────────────┐
│  人类解决问题的认知层次                                       │
│                                                            │
│  Level 1: 反射（Reflex）                                   │
│  → 看到红灯就停下                                           │
│  → 对应: Standard Prompting（直接回答）                     │
│                                                            │
│  Level 2: 边想边做（Think-then-Act）                        │
│  → 做菜时一边看菜谱一边操作                                  │
│  → 对应: ReAct（交替 Thought-Action）                       │
│                                                            │
│  Level 3: 先规划再执行（Plan-Execute-Adapt）                 │
│  → 装修房子前先画设计图，施工中根据实际调整                    │
│  → 对应: Pre-Act（多步计划 + 增量修正）                      │
│                                                            │
│  Level 4: 多层级规划（Hierarchical Planning）                │
│  → 项目经理拆解季度目标为月/周/日任务                         │
│  → 对应: 未来方向（MCTS + Agent 分层协作）                    │
└────────────────────────────────────────────────────────────┘
```

### 2. "小模型 + 好方法 > 大模型 + 旧方法"的实践意义

这是论文最具产业价值的结论。对企业部署 Agent 系统来说：

```
┌─────────────────┬─────────────────┬───────────────────┐
│                 │  GPT-4 + ReAct  │  Llama70B 微调    │
│                 │                 │  + Pre-Act        │
├─────────────────┼─────────────────┼───────────────────┤
│  目标完成率      │  32%            │  83%  (+159%)     │
│  推理延迟        │  高（API 调用）  │  低（本地推理）    │
│  单次调用成本     │  ~$0.01-0.03    │  ~$0.001          │
│  数据隐私        │  数据出境       │  本地部署          │
│  可定制性        │  只能调 Prompt   │  可针对领域微调    │
└─────────────────┴─────────────────┴───────────────────┘

  结论: 在 Agent 场景中，"好的推理格式 + 领域微调"
  比"堆模型参数量"更有效、更经济。
```

### 3. 评估方法的启发

论文提出的两级评估框架揭示了一个容易被忽视的问题：**逐步正确 ≠ 整体成功。**

GPT-4 + ReAct 在 Level 1（逐步评估）的 Action Recall 有 0.443，看起来还行。但在 Level 2（端到端评估）的目标完成率只有 0.32。这意味着：即使单步动作经常是对的，但如果缺乏全局规划，Agent 仍然无法可靠地完成复杂任务。

这对 Agent 系统的评估提出了重要启示：**不能只测单步准确率，必须测端到端完成率。**

## Pre-Act 的局限性

*   **推理开销增大：** 每步都生成完整计划意味着更多的 token 生成，推理延迟和成本会增加。论文未量化这部分开销。
*   **标注成本：** Pre-Act 格式的训练数据需要人工为每一步撰写详细推理，标注成本高于 ReAct。
*   **评估依赖 LLM-as-Judge：** Level 2 评估使用 GPT-4 作为裁判，本身存在判断波动性，论文也承认了这一点。
*   **场景覆盖有限：** 实验主要在任务型对话场景中验证，尚未涵盖代码生成、复杂推理等 Agent 应用场景。

---

### 总结

Pre-Act 的核心贡献可以概括为三个层面：

1. **方法层面**：将 ReAct 的"单步推理"升级为"多步规划 + 增量修正"，让 Agent 具备全局视野和动态适应能力。
2. **工程层面**：通过课程学习微调策略，使 Llama 70B 在 Agent 任务上超越 GPT-4，证明了"小模型 + 好方法"的可行性。
3. **评估层面**：提出两级评估框架（逐步 + 端到端），为 Agent 系统的评估提供了更完整的方法论。

从 Agent 发展脉络来看，Pre-Act 是 ReAct → Plan-and-Execute 演进中的重要一环，它回答了一个实用的问题：**如何让 Agent 在复杂多步任务中不迷路？答案是给它一张地图，而不是只给一个指南针。**
