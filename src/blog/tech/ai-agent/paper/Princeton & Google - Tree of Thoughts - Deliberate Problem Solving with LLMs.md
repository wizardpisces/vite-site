*论文发布时间：2023-05-17*

Princeton 和 Google DeepMind 联合发布的 **Tree of Thoughts（ToT）**（论文标题为《[Tree of Thoughts: Deliberate Problem Solving with Large Language Models](https://arxiv.org/abs/2305.10601)》）是 LLM 推理范式的里程碑式工作。
简单来说，这篇论文提出了一种**将经典搜索算法（BFS/DFS）与 LLM 的自我评估能力相结合**的推理框架——让模型不再沿单一路径从左到右解码，而是像人类一样在**思维树**上进行探索、评估、前瞻和回溯，实现"深思熟虑"式的问题求解。

## ToT 解决了什么问题？

**核心问题：LLM 的推理是"单行道"的——一旦走错，无法回头。**

![Figure 1: 各种 LLM 问题求解方法的架构对比。每个矩形框代表一个"思维"（thought），即作为问题求解中间步骤的连贯语言序列。(a) IO 直接输入输出；(b) CoT 单链推理；(c) CoT-SC 多链投票；(d) ToT 树状探索——支持多路径生成、评估与回溯。](https://arxiv.org/html/2305.10601v2/x1.png)

无论是标准的 IO prompting 还是 Chain-of-Thought（CoT），LLM 的推理本质上都是**自回归的单链生成**——每个 token 只能依赖前面的所有 token，一旦某一步推理出错，后续所有步骤都在错误的基础上继续，且无法回溯修正：

```
【单链推理的根本缺陷】

  IO prompting（无中间步骤）:
  输入 x ────────────────→ 输出 y
  "4 9 10 13"               "4+9+10+13=36" ✗  走了就走了

  Chain-of-Thought（有中间步骤，但仍是单链）:
  输入 x → z₁ → z₂ → z₃ → 输出 y
  "4 9 10 13"  "4+9=13"  "13-10=3"  "3×13=39" ✗  第一步走偏，后面全错

  CoT-SC（多条独立链，投票选最佳）:
  输入 x → 链₁ → y₁ ╲
         → 链₂ → y₂ ─→ 投票 → y_best
         → 链₃ → y₃ ╱
  ↑ 每条链内部仍然是单链，只是最后多数表决
  ↑ 链与链之间不共享中间步骤的信息
```

这在需要**探索、规划或策略性前瞻**的任务中是致命的。以 Game of 24（用 4 个数和四则运算凑出 24）为例，GPT-4 用 CoT 仅解出 **4%** 的测试题——因为约 60% 的 CoT 采样在生成前三个词（即第一步运算）时就已经走入了死胡同。

**对比人类解题过程：**

```
人类解 Game of 24 (4, 9, 10, 13)：

  "让我想想... 4+9=13？不行，两个 13 做不出 24"
      │
      └── 回溯 ──→ "那试试 13-9=4... 现在有 4,4,10"
                         │
                         ├── "4+4=8, 8+10=18... 不行"
                         │
                         └── 回溯 ──→ "10-4=6, 4×6=24！✓"

  关键能力：
  ✓ 在每一步考虑多种可能性
  ✓ 评估当前状态是否有前途
  ✓ 发现走不通时果断回溯
  ✓ 全局搜索而非局部贪心
```

LLM 的自回归解码缺乏这种**"退一步海阔天空"**的能力，而 ToT 正是要补上这块缺失。

## 这个问题真实存在吗？

**真实存在，而且在需要搜索和规划的任务中是 LLM 能力的硬天花板。**

* **数学推理：** Game of 24 要求在组合空间中搜索有效的运算路径。CoT 的成功率仅 4%，即使采样 100 条独立链（CoT-SC, k=100）也只有 9%。
* **创意写作：** 要求生成一段**结尾分别对应 4 个随机句子**的 4 段式文章。这需要全局规划段落间的衔接逻辑，CoT 的单链模式难以兼顾整体连贯性。
* **填字游戏：** 5×5 Mini Crosswords 涉及 10 条线索的交叉约束，每填一个词都可能影响其他词的候选。CoT 的词级成功率不到 16%。

论文以心理学的**"双系统理论"（Dual Process Theory）**为理论基础：
* **System 1（快思考）：** 自动的、无意识的——对应 LLM 的自回归解码。
* **System 2（慢思考）：** 刻意的、有意识的——对应 ToT 的搜索与评估。

LLM 目前的推理主要是 System 1 模式，ToT 则为其增添了 System 2 的"深思熟虑"能力。

## 它是如何解决的？

ToT 将问题求解建模为一棵**搜索树**，其中每个节点是一个**状态** $s = [x, z_{1 \cdots i}]$（输入加上到目前为止的思维序列），每条边代表一个**思维步骤**。整个框架需要回答四个核心问题：

### 1. 思维分解（Thought Decomposition）

CoT 将思维视为一整段连续文本，不做显式分解。ToT 则根据任务特性将中间过程**分解为适当粒度的思维步骤**：

```
┌─────────────┬──────────────────┬───────────────────┐
│  任务         │  思维粒度         │  思维步数           │
├─────────────┼──────────────────┼───────────────────┤
│  Game of 24  │  一行等式          │  3 步              │
│              │  如 "13-9=4"      │  (三次运算)         │
├─────────────┼──────────────────┼───────────────────┤
│  创意写作     │  一段写作计划       │  1 步              │
│              │  如 "1. 引入一本   │  (先规划后写作)      │
│              │  联系...的书"      │                    │
├─────────────┼──────────────────┼───────────────────┤
│  填字游戏     │  一个词            │  5-10 步（可变）    │
│              │  如 "h1. SHOWN"   │  (逐词填入)         │
└─────────────┴──────────────────┴───────────────────┘

分解的原则：
  "小"到足以让 LM 生成多样的候选（生成一整本书太"大"了）
  "大"到足以让 LM 评估其前景（生成一个 token 太"小"了）
```

### 2. 思维生成（Thought Generator）

给定当前状态 $s$，生成 $k$ 个候选思维步骤。两种策略：

```
策略 A: 独立采样（适用于丰富的思维空间，如创意写作）
  ───────────────────────────────────
  z⁽¹⁾ ~ p_θ(z_{i+1} | s)    ← 独立采样 k 次
  z⁽²⁾ ~ p_θ(z_{i+1} | s)      （i.i.d.，每次独立）
  ...
  z⁽ᵏ⁾ ~ p_θ(z_{i+1} | s)    → 天然保证多样性

策略 B: 顺序提议（适用于受约束的思维空间，如 Game of 24）
  ───────────────────────────────────
  [z⁽¹⁾, z⁽²⁾, ..., z⁽ᵏ⁾] ~ p_θ(z_{i+1}^{(1..k)} | s)
                                 ← 一次生成所有候选
                                   在同一上下文中提议
                                 → 避免生成重复的选项
```

### 3. 状态评估（State Evaluator）

![Figure 2: Game of 24 中的 ToT。LM 被提示进行 (a) 思维生成：给定剩余数字，提议可能的下一步运算；(b) 状态评估：判断当前剩余数字是 sure/maybe/impossible 能凑出 24。](https://arxiv.org/html/2305.10601v2/x2.png)

这是 ToT 最核心的创新——**让 LM 自我评估中间状态的前景**，充当搜索的启发式函数：

```
策略 A: 独立评值（适用于可量化的任务，如 Game of 24）
  ───────────────────────────────────
  状态 s = "输入 4 9 10 13; 已做 13-9=4, 剩余 4 4 10"

  LM 评估: "4,4,10 能凑出 24 吗？"
    → "sure"   (如 5+5+14=24 这种显然可以)
    → "maybe"  (不确定，保留探索)
    → "impossible" (如 1,2,3 太小了，剪枝)

  对每个候选状态独立评估，可多次采样取平均

策略 B: 投票比较（适用于难以直接评分的任务，如创意写作）
  ───────────────────────────────────
  给定候选状态集合 S = {s₁, s₂, ..., sₖ}

  LM 投票: "分析以下方案，哪个最有前途？"
    → 投票 5 次，取多数票
    → 类似于"逐步自我一致性"
```

**这一设计是前所未有的**——以往的搜索启发式要么是人工编写的规则（如 Deep Blue），要么是训练出来的模型（如 AlphaGo），ToT 提出了**第三条路：让 LM 用自然语言推理来充当启发式函数**。

### 4. 搜索算法（Search Algorithm）

在思维树上执行系统化搜索，论文探索了两种经典算法。

**BFS 用于创意写作的实例：**

![Figure 4: 创意写作任务中的 deliberate search 步骤。给定输入约束，LM 先采样 5 种不同的写作计划，再投票 5 次选出最佳计划。多数票的方案被用于后续写作，同样采用 sample-vote 流程。](https://arxiv.org/html/2305.10601v2/x4.png)

**DFS 用于填字游戏的实例：**

![Figure 6: Mini Crosswords 中的 ToT。(a) 如何在优先队列中提议和聚合思维以进行 DFS；(b) 如何基于剩余线索的可填写性评估状态，若任何线索被判定为不可能填写则剪枝，DFS 回溯到父状态。](https://arxiv.org/html/2305.10601v2/x6.png)

```
【BFS — 广度优先搜索（用于 Game of 24 和创意写作）】

  步骤 t=1:                步骤 t=2:              步骤 t=3:
  从输入生成 k 个候选       每个保留节点再生成 k 个   继续...
  ┌─ z₁ (sure)   ← 保留   ┌─ z₁₁ (sure) ✓
  ├─ z₂ (maybe)  ← 保留   ├─ z₁₂ (impossible) ✗
  ├─ z₃ (impossible) ✗     ├─ z₂₁ (maybe) ← 保留
  ├─ z₄ (sure)   ← 保留   └─ ...
  └─ z₅ (maybe)  ← 保留

  每步保留最佳 b 个状态（b=5 for Game of 24）
  适合树深度有限（T≤3）的场景

【DFS — 深度优先搜索（用于填字游戏）】

  优先深入最有前途的分支
  ┌─ h1=SHOWN ✓
  │  ├─ v1=SWEAR ✓
  │  │  ├─ h2=WIRRA → v2 无解 → impossible! → 回溯 ↩
  │  │  │
  │  │  └─ h2=WIRED ✓
  │  │     └─ ...继续深入
  │  │
  │  └─ v1=STAIR ✓
  │     └─ ...
  │
  └─ h1=TASKS ✓
     └─ ...

  遇到 "impossible" 就剪枝回溯
  搜索步数限制为 100 步
  适合深度可变、需要回溯的场景
```

### 5. 完整数据流：以 Game of 24 为例

```
输入: "4 9 10 13"

━━ 步骤 1: 生成 + 评估第一步运算 ━━

  LM 生成 k 个候选:
    z₁: "13 - 9 = 4  (剩余: 4, 4, 10)"
    z₂: "10 - 4 = 6  (剩余: 6, 9, 13)"
    z₃: "4 + 9 = 13  (剩余: 10, 13, 13)"
    z₄: "9 × 4 = 36  (剩余: 10, 13, 36)"  ← 太大了
    z₅: "13 + 10 = 23 (剩余: 4, 9, 23)"

  LM 评估每个候选（采样 3 次取多数）:
    z₁: "sure"        ← 保留 ✓
    z₂: "maybe"       ← 保留 ✓
    z₃: "maybe"       ← 保留 ✓
    z₄: "impossible"  ← 剪枝 ✗
    z₅: "maybe"       ← 保留 ✓

  保留 top-5（此例保留 4 个）

━━ 步骤 2: 对每个保留状态重复生成 + 评估 ━━

  以 z₁ (剩余 4, 4, 10) 为例:
    z₁₁: "10 - 4 = 6  (剩余: 4, 6)"   → "sure" ✓
    z₁₂: "4 + 4 = 8  (剩余: 8, 10)"   → "maybe"
    z₁₃: "4 × 4 = 16 (剩余: 10, 16)"  → "maybe"
    ...

━━ 步骤 3: 最后一步运算 ━━

  以 z₁₁ (剩余 4, 6) 为例:
    "4 × 6 = 24" → "sure" ✓  ← 找到解！

最终输出: "(10 - 4) × (13 - 9) = 24"
```

## 实验结果

![Figure 3: Game of 24 (a) 规模分析：ToT 随搜索节点数增长的成功率远优于 IO/CoT 的独立采样；(b) 错误分析：约 60% 的 CoT 采样在第一步就已失败，而 ToT (b=5) 在每一步都保持了极低的失败率。](https://arxiv.org/html/2305.10601v2/x3.png)

| 任务 | 方法 | 成功率 |
|------|------|--------|
| Game of 24 | IO prompt | 7.3% |
| | CoT prompt | 4.0% |
| | CoT-SC (k=100) | 9.0% |
| | **ToT (b=5)** | **74%** |
| 创意写作 | IO | 6.19 分 |
| | CoT | 6.93 分 |
| | **ToT** | **7.56 分** |
| 填字游戏 | CoT (词级) | 15.6% |
| | **ToT (词级)** | **60%** |

![Figure 5: 创意写作结果。(a) GPT-4 自动评分：ToT (7.56) 显著优于 IO (6.19) 和 CoT (6.93)；(b) 人类评审：100 对比较中 ToT 胜出 41 次 vs CoT 胜出 21 次。](https://arxiv.org/html/2305.10601v2/x5.png)

关键发现：
* **Game of 24：** ToT 将成功率从 4%（CoT）提升到 74%，提升了 **18.5 倍**。
* **创意写作：** 人类评审中，ToT 在 100 对比较中胜出 41 次 vs CoT 胜出 21 次。
* **填字游戏：** ToT 的词级成功率是 CoT 的近 **4 倍**，并成功完整解出 4/20 个游戏。
* **消融实验：** 去掉剪枝（-prune）或回溯（-backtrack）都导致显著性能下降，证明了搜索机制各组件的必要性。

## 从 Engram 视角看 ToT：System 1 与 System 2 的互补

ToT 论文以 Kahneman 的**双系统理论**为出发点，而这恰好与 Engram 的设计哲学形成了一组精妙的互补。把两篇论文放在一起看，会发现它们分别增强了大模型的两种截然不同的认知能力：

```
┌─────────────────────────────────────────────────────────────────┐
│              LLM 的"双系统"增强路线图                              │
│                                                                  │
│  ┌─── System 1: 快思考 ───────────────────────────────────┐     │
│  │                                                        │     │
│  │  DeepSeek Engram (2026)                                │     │
│  │  ─────────────────────────                             │     │
│  │  增强什么？→ 静态知识的即时检索                          │     │
│  │  怎么增强？→ O(1) 哈希查表 + 上下文门控                  │     │
│  │  在哪增强？→ 模型架构内部（浅层 FFN 旁，DRAM 存储）      │     │
│  │  何时起效？→ 训练时学习 + 推理时查表                     │     │
│  │                                                        │     │
│  │  类比：给大脑装了一本"可微分词典"                        │     │
│  │  → 看到 "Princess of Wales" 直接查出实体知识              │     │
│  │  → 不需要层层神经网络推演                                │     │
│  │                                                        │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                  │
│  ┌─── System 2: 慢思考 ───────────────────────────────────┐     │
│  │                                                        │     │
│  │  Princeton ToT (2023)                                  │     │
│  │  ─────────────────────────                             │     │
│  │  增强什么？→ 复杂问题的深思熟虑求解                      │     │
│  │  怎么增强？→ 树搜索（BFS/DFS）+ LM 自评估               │     │
│  │  在哪增强？→ 模型外部（推理时的 prompting 框架）          │     │
│  │  何时起效？→ 仅推理时（不修改模型参数）                   │     │
│  │                                                        │     │
│  │  类比：给大脑装了一套"搜索与回溯系统"                    │     │
│  │  → 解数学题时在草稿纸上尝试多条路径                      │     │
│  │  → 走不通就退回重新来                                    │     │
│  │                                                        │     │
│  └────────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
```

### 1. 架构层 vs 推理层：两个完全不同的干预层次

这是两篇论文最根本的差异。Engram 改变的是**模型本身的结构**，而 ToT 改变的是**模型被使用的方式**：

```
┌──────────────┬──────────────────────────┬──────────────────────────┐
│              │  Engram                   │  ToT                      │
├──────────────┼──────────────────────────┼──────────────────────────┤
│  干预层次     │  架构层（训练+推理）       │  推理层（仅推理）           │
│  是否改模型   │  ✓ 新增查找表+门控模块     │  ✗ 完全不改模型             │
│  是否需训练   │  ✓ 端到端联合训练          │  ✗ 即插即用                │
│  参数变化     │  重新分配稀疏参数          │  零参数变化                │
│  计算位置     │  GPU + DRAM              │  多次 LM API 调用          │
│  额外成本     │  DRAM 存储 + PCIe 带宽    │  5-100x token 生成量       │
└──────────────┴──────────────────────────┴──────────────────────────┘
```

这意味着两者是**天然可叠加的**——你可以在一个 Engram 增强的模型上运行 ToT 推理框架，同时获得两种增强。

### 2. "记忆"与"搜索"的分工

从认知科学的角度看，Engram 和 ToT 恰好覆盖了人类问题求解的两个互补环节：

```
人类解决复杂问题的过程：

  "法国大革命发生在哪一年？它如何影响了后来的拿破仑战争？"

  步骤 1 — 事实检索（System 1 / Engram 的领域）
  ─────────────────────────────────────────
  "法国大革命... 1789 年"  ← 即时回忆，不需要推理
  "拿破仑战争... 1803-1815 年"  ← 同上
  → Engram 的 O(1) 哈希查表就是在做这件事

  步骤 2 — 逻辑推理（System 2 / ToT 的领域）
  ─────────────────────────────────────────
  "大革命推翻了君主制 → 拿破仑在混乱中崛起 → 
   但他的军事扩张 → 触发了反法同盟 → 导致了..."
  → 这需要在多条因果链中探索、评估、选择
  → ToT 的树搜索就是在做这件事
```

Engram 论文的消融实验已经验证了这一分工：关闭 Engram 后，事实类任务（TriviaQA）暴跌到 29%，但阅读理解（C3）仅下降到 93%——**说明事实检索和推理确实是不同的认知能力**。ToT 增强的恰恰是 Engram 不擅长的后者。

### 3. 确定性 vs 探索性：两种截然不同的"查找"

两篇论文虽然都涉及"搜索"，但它们的搜索性质完全不同：

```
Engram 的"搜索"：确定性查找
──────────────────────────
  输入 N-gram → 固定哈希函数 → 确定的地址 → 确定的向量
  
  特点：
  ✓ 严格 O(1)，没有任何不确定性
  ✓ 同一个输入永远返回同一个结果
  ✓ 不需要评估、不需要回溯
  ✓ 适合已经学过的、确定性的知识
  
  类比：查字典——翻到"A"开头的页面就能找到"Apple"

ToT 的搜索：探索性搜索
──────────────────────────
  当前状态 → 生成多个候选 → 评估 → 选择/回溯

  特点：
  ✓ O(b^T) 最坏情况，但通过剪枝大幅减少
  ✓ 同一个输入可能走不同的路径
  ✓ 核心是评估和回溯机制
  ✓ 适合未知的、需要发现的解题路径
  
  类比：走迷宫——需要尝试、碰壁、退回、换路
```

### 4. 浅层注入 vs 全局重组：对信息流的不同影响

Engram 在模型**浅层（Layer 2-4）**注入静态知识，改变的是**局部的隐藏状态**；ToT 在模型**外部**重组推理路径，改变的是**全局的 token 序列**：

```
Engram 的信息流改变（模型内部，微观）：
──────────────────────────────────
  Layer 1 输出 h_t
       │
       + Engram 查表结果 Y_t    ← 浅层注入，一次性
       │
  Layer 2 Attention（输入已增强）
       │
  Layer 3-30（正常计算）
       │
  输出

  → 信息流被"提前注入"了事实知识
  → 深层网络被释放去做推理
  → 论文结论：Engram Layer 5 ≈ MoE Layer 12

ToT 的信息流改变（模型外部，宏观）：
──────────────────────────────────
  问题 x
       │
   ┌───┴───┐
  LM 生成  LM 生成  LM 生成     ← 多次调用 LM
  思维 z₁  思维 z₂  思维 z₃
   │        │        │
  LM 评估  LM 评估  LM 评估     ← 再次调用 LM
  "sure"   "impos." "maybe"
   │                  │
   └────┬─────────────┘
        │ 选择最佳
     下一轮生成...

  → 信息流被"树状重组"
  → 每个节点都是一次完整的 LM forward pass
  → 用多次 forward pass 换取更高的推理质量
```

### 5. 如果 Engram 遇上 ToT：理想的认知架构？

将两者结合的假想模型可能会获得非常互补的能力增强：

```
假想的 "Engram + ToT" 系统：

  问题: "用 1789, 4, 7, 14 凑出 24"

  ━━ Engram 层面（每次 LM 调用内部自动生效）━━
  
  模型处理 "1789" 时：
    Engram 查表 → "1789 = 法国大革命年份"
    门控 α_t → 当前是数学任务，α_t ≈ 0.05 → 忽略
    → Engram 自动判断这条记忆与当前语境无关

  模型处理算术步骤时：
    Engram 查表 → 常见算术模式
    门控 α_t → 数学语境匹配，α_t ≈ 0.8 → 采纳
    → Engram 辅助快速识别可行的运算模式

  ━━ ToT 层面（在多次 LM 调用之间协调）━━
  
  生成候选: "1789 ÷ 7 = 255.57..." → 评估: impossible ✗
  生成候选: "14 - 7 = 7, 1789..." → 评估: impossible ✗
  生成候选: "14 × (7 - 4) - 1789..." → 评估: impossible ✗
  回溯 → 重新思考...
  生成候选: "7 × 4 - 14 ÷ (1789-1789)..." → 除零！✗
  ...

  → Engram 在每次"快思考"中提供知识支撑
  → ToT 在"慢思考"层面协调多次尝试
```

## 关键洞察

### 1. 启发式函数的"涌现"

ToT 最深刻的贡献不是树搜索本身（这是 1950 年代就有的老思想），而是**让 LM 用自然语言推理来充当启发式函数**。传统搜索需要人工设计评估函数（Deep Blue 的棋盘评估）或训练专用模型（AlphaGo 的价值网络），而 ToT 发现：**足够强大的 LM 已经涌现出了评估中间状态的能力**，只需通过恰当的 prompt 就能激活。

这与 Engram 门控机制的 $\alpha_t$ 形成有趣的呼应——两者都是"评估信息是否有用"的机制，只是一个在模型内部（可微分的标量门控），一个在模型外部（自然语言推理的启发式）。

### 2. 通用性与模块化

ToT 的四个组件（思维分解、生成、评估、搜索）完全正交，可以独立替换：

* IO / CoT / CoT-SC 都是 ToT 的特例（深度/宽度为 1 的退化树）
* 可以为不同任务定制不同的分解粒度和搜索策略
* 不需要额外训练，即插即用

这种模块化设计和 Engram 的"计算与存储解耦"哲学一脉相承——**好的系统设计应该让不同能力独立扩展**。

### 3. 成本与效率的权衡

ToT 的主要代价是**推理时的计算成本**——解一道 Game of 24 需要约 5500 个 completion token（相当于 ~100 次 CoT 的 token 量），API 成本约 $0.74/题。而 Engram 的代价是**存储成本**——需要额外的 DRAM 来存放嵌入表，但推理时的 FLOPs 不增加。

```
两种增强方式的成本模型：

  Engram: 一次性成本（训练+部署）
  ─────────────────────────────
  训练: 端到端训练，无额外训练成本
  部署: DRAM 存储（~22GB/层）+ PCIe 带宽
  推理: 严格 O(1) 查表，FLOPs 不变
  → 边际成本 ≈ 0（每次推理几乎免费）

  ToT: 每次推理都有额外成本
  ─────────────────────────────
  训练: 无（使用现成 LM）
  部署: 无额外基础设施
  推理: 5-100x 的 token 生成量
  → 边际成本高（每次推理都要多次 LM 调用）
```

这暗示了一个研究方向：**能否将 ToT 的搜索能力"内化"到模型架构中？** 就像 Engram 将 RAG 的检索能力内化为可微分的查找表一样，或许未来的模型可以在架构层面实现某种"内置搜索"——在训练时学会在隐藏状态空间中进行类似树搜索的探索。事实上，DeepSeek-R1 的强化学习训练已经在朝这个方向迈出了一步。

## 冷思考：CoT-SC 比 ToT 更实用？

ToT 的论文写得漂亮，但实际落地比 CoT-SC 难得多。核心原因就一个字：**评估不靠谱**。

### CoT-SC 为什么实用？

因为它的假设极其简单——"正确答案出现的概率最高"。采样 N 次 → 投票 → 完事。没有任何额外的 prompt 设计，完全可并行，实现 10 行代码搞定。

### ToT 的"美丽陷阱"

ToT 的整个框架都建立在一个脆弱的前提上：**LM 能准确评估中间状态的好坏**。但实际上：

1. **评估 prompt 本身就很难设计。** 论文里 Game of 24 的 "sure/maybe/impossible" 能用，是因为数学题有确定性答案。换成开放式任务（写代码、做决策），LM 给中间状态打分本身就不靠谱——论文在 Crosswords 实验里也承认了，正确解有时反而被评估器剪掉了。

2. **搜索策略要 per-task 定制。** 论文三个任务分别用了不同的思维粒度、不同的生成策略（i.i.d. vs propose）、不同的评估策略（value vs vote）、不同的搜索算法（BFS vs DFS）。面对一个新任务，怎么选？"模块化"既是优点也是负担。

3. **成本不可忽视。** 论文自己算的账：ToT 解一道 Game of 24 花 $0.74，CoT best-of-100 才 $0.47，而且 CoT 是可以无脑并行的，ToT 的 BFS 每一层都要等上一层评估完。

### 真正的演进方向

自 2023 年 ToT 发表以来，业界的实际选择验证了这个判断——与其在推理时做外部搜索，不如**把搜索能力训练进模型内部**：

```
ToT 的理想 vs 现实：

  ToT 的路线（外部搜索）:
  ────────────────────────
  现成 LM + 外部树搜索框架 + 手工设计的评估 prompt
  → 灵活但脆弱，每个任务都要重新调
  → 评估器本身不可靠，搜索可能越搜越偏
  → 成本高（5-100x token）

  业界实际走的路线（内部搜索）:
  ────────────────────────
  OpenAI o1/o3、DeepSeek-R1:
    通过 RL 训练模型在内部 token 流中自动"探索-评估-回溯"
    → 本质上是把 ToT 的树搜索"内化"成了模型的思维习惯
    → 不需要外部框架，不需要手工设计评估 prompt

  Best-of-N（本质就是 CoT-SC）:
    至今仍是生产环境中最常用的推理增强方法
    → 简单、可控、好扩展、可并行
```

#### "内部搜索"具体是怎么做到的？

回顾一下 ToT 的"外部搜索"做了三件事：**生成**多个候选思路（树的分支）→ **评估**每个思路好不好（用 LLM 打分）→ **回溯**到之前的节点换一条路。这三步都由外部 Python 框架驱动，模型本身并不知道自己在"搜索"。

o1/R1 的关键思路是：**用 RL 训练让模型在自己的 token 输出中自发涌现出这三种行为**。

**训练机制：** 给模型一个难题（比如数学题），让它输出一段很长的"思考过程"（extended chain-of-thought），然后只根据最终答案的对错给奖励（reward）。RL 不关心中间过程——模型为了拿到更高奖励，自然会演化出"试错-验证-回退"的策略，因为只走一条路容易错，而在 token 流中多尝试几条路并自我纠正，答对概率就高，奖励就大。

**涌现出的行为模式**（在 R1 的 CoT 中可以直接观察到）：

```
内部 token 流示例（以 Game of 24 为例，简化展示）：

"让我试试方法A：先算 4×6=24，但还剩 1 和 9 没用...不对。  ← 探索 + 评估
 等等，换个思路，如果先用 8-4=4...                         ← 回溯
 4×(3+3)=24？不对只有一个 3...                             ← 再次评估
 重新想，(10-4)×(9-5)... 让我验算：6×4=24，对了！"         ← 探索新分支 + 确认
```

对比 ToT 的外部搜索和 o1/R1 的内部搜索：

| | ToT（外部搜索） | o1/R1（内部搜索） |
|---|---|---|
| 搜索控制 | 外部 Python 框架显式驱动 | 模型在 token 流中自主完成 |
| 评估器 | 单独的 LLM prompt 打分 | 模型用自然语言自我验证 |
| 回溯 | 程序恢复到之前的状态节点 | 模型说"不对，换个方向" |
| 任务适配 | 每个任务需手工设计评估 prompt | RL 训练后可泛化到新任务 |
| 训练成本 | 不需要额外训练 | 需要大量 RL 训练 |

一个直观的类比：ToT 像是一个人拿着纸笔显式画决策树来解题，o1/R1 则像一个经验丰富的人在脑子里自动权衡多条路径——后者更自然、更高效，但需要大量练习（RL 训练）才能习得这种"内化的搜索直觉"。

**结论：ToT 的理论贡献很大**（把推理建模为搜索问题，首次让 LM 自己当启发式函数），但**实践上 CoT-SC 确实是更好的起点**。ToT 更像是一篇"指明方向"的论文——它告诉我们 LM 推理应该往搜索的方向走，但最终的答案不是在推理时外挂一棵树，而是让模型自己学会"长出树来"。

## 总结

ToT 和 Engram 分别从**推理层**和**架构层**增强了 LLM 的能力，恰好覆盖了人类认知的两个互补系统：

1. **Engram（System 1）：** 用 O(1) 查表增强**快速知识检索**——"我记得这个事实"。
2. **ToT（System 2）：** 用树搜索增强**深思熟虑推理**——"让我仔细想想这个问题"。
3. **两者的结合**暗示了未来 AI 系统的理想形态：既有闪电般的知识调取能力，又有系统化的搜索与规划能力——正如人类大脑同时具备直觉反应和理性思考。
