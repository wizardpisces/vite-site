*论文发布时间：2026-02-25*

DeepSeek 联合北大、清华发布的 **DualPath**（论文标题为《[DualPath: Breaking the Storage Bandwidth Bottleneck in Agentic LLM Inference](https://arxiv.org/abs/2602.21548)》）直击当下 AI Agent 推理的核心瓶颈。
简单来说，这篇论文发现在多轮 Agent 推理场景下，系统性能的瓶颈**不是 GPU 算力，而是从硬盘加载 KV-Cache 的带宽**。DualPath 通过让"闲着的"解码引擎帮忙搬运数据，实现了最高 1.87 倍的吞吐提升。

## 先认识几个关键硬件概念

DualPath 是一篇**系统论文**，围绕数据中心硬件展开。正文中会反复出现以下术语，先用一张图建立直觉：

```
┌──────────────── AI 服务器节点 ────────────────────┐
│                                                    │
│  ┌─────┐ ┌─────┐ ┌─────┐         ┌─────┐         │
│  │GPU 0│ │GPU 1│ │GPU 2│  ...    │GPU 7│         │
│  └──┬──┘ └──┬──┘ └──┬──┘         └──┬──┘         │
│     │       │       │               │              │
│  ┌──┴──┐ ┌──┴──┐ ┌──┴──┐         ┌──┴──┐         │
│  │CNIC │ │CNIC │ │CNIC │  ...    │CNIC │ ×8      │
│  └──┬──┘ └──┬──┘ └──┬──┘         └──┬──┘         │
│     └───────┴───┬───┴───────────────┘              │
│                 │                                   │
│     ┌───────────┴──────────────┐                   │
│     │  InfiniBand 计算网络      │                   │
│     │  (GPU 间高速互联)         │                   │
│     └──────────────────────────┘                   │
│                                                    │
│              ┌──────┐                              │
│              │ SNIC │ ×1（整个节点共享）             │
│              └──┬───┘                              │
│     ┌───────────┴──────────────┐                   │
│     │  存储网络                 │                   │
│     │  (连接 SSD 分布式存储)    │                   │
│     └──────────────────────────┘                   │
└────────────────────────────────────────────────────┘
```

| 缩写 | 全称 | 一句话解释 |
|---|---|---|
| **SNIC** | Storage NIC（存储网卡） | 连接 SSD 存储的网卡，每个**节点**只有 1 张（400Gbps），所有 GPU 共享。**本文的瓶颈所在** |
| **CNIC** | Compute NIC（计算网卡） | 连接 GPU 间高速网络的网卡，每个 **GPU** 配 1 张（400Gbps），总带宽是 SNIC 的 8 倍 |
| **RDMA** | Remote Direct Memory Access（远程直接内存访问） | 让网卡**绕过 CPU/操作系统**直接读写远端机器内存的技术，延迟仅 ~1μs |
| **InfiniBand** | — | AI 数据中心主流的高速网络协议，CNIC 之间通过它通信，支持 RDMA |
| **SSD** | Solid State Drive（固态硬盘） | 持久化存储，存放 Agent 的海量历史 KV-Cache |
| **DRAM** | Dynamic RAM（主内存） | 服务器的"内存条"，比 SSD 快 10 倍但贵 10 倍 |
| **HBM** | High Bandwidth Memory（GPU 显存） | GPU 自带的显存，最快但容量最小 |

> 记住一个核心矛盾就够了：**SNIC 只有 1 张（慢），CNIC 有 8 张（快）。DualPath 的本质就是借 CNIC 的路，帮 SNIC 分担压力。**

## DualPath 解决了什么问题？

![Figure 1: 现有瓶颈（左）与 DualPath 方案（右）。传统架构中 Prefill 侧 SNIC 饱和，而 Decode 侧 SNIC 空闲；DualPath 通过双路径加载池化所有存储带宽。](https://arxiv.org/html/2602.21548v1/x1.png)

**核心问题：Agent 推理中，Prefill 引擎的存储网卡被"撑爆"了，而 Decode 引擎的网卡却在"摸鱼"。**

当前主流的 LLM 推理系统采用 **PD 分离架构（Prefill-Decode Disaggregation）**。

虽然 GPT、DeepSeek 等模型都是 **decoder-only** 架构（没有编码器），但它们在推理时天然存在两个**计算特性截然不同**的阶段：

* **Prefill（预填充）阶段：** 把用户输入的所有 token **一次性并行处理**，为每个 token 生成对应的 Key 和 Value 向量（即 KV-Cache）。这个阶段是**计算密集型**——大量 token 可以并行做矩阵乘法，GPU 的算力是瓶颈。
* **Decode（解码）阶段：** 拿着 Prefill 生成的 KV-Cache，**逐个 token** 自回归生成。每一步只产出一个 token，却需要读取整个 KV-Cache 来计算注意力。这个阶段是**访存密集型**——GPU 的显存带宽是瓶颈，算力大量空闲。

两个阶段对硬件的需求完全相反：Prefill 需要高算力，Decode 需要高带宽。混在一起跑，就像让短跑运动员和马拉松选手共用一条赛道——谁都跑不好。**PD 分离的本质就是把这两个阶段拆到不同的 GPU 集群上，各自优化：**

```
用户输入（数千~数万 token）
            ↓
┌──────────────────────────┐
│   Prefill GPU 集群        │  ← 计算密集：并行处理所有输入 token
│   生成完整的 KV-Cache     │     GPU 算力拉满
└──────────┬───────────────┘
           │  通过 RDMA 高速网络传输 KV-Cache
           ↓
┌──────────────────────────┐
│   Decode GPU 集群         │  ← 访存密集：逐 token 自回归生成
│   逐 token 生成回答       │     GPU 显存带宽拉满
└──────────┬───────────────┘
           │  生成的 KV-Cache 持久化到 SSD
           ↓
┌──────────────────────────┐
│   分布式 SSD 存储（3FS）   │  ← 下一轮对话时，Prefill 从这里加载历史 KV-Cache
└──────────────────────────┘
```

为了支持 Agent 的多轮对话，每轮 Decode 产生的 KV-Cache 会写入外部 SSD 存储。下一轮对话时，Prefill 引擎从 SSD 加载历史 KV-Cache（命中部分），只需对新追加的 token 做计算。

但这个架构有一个结构性缺陷——**存储带宽的不对称**：

```
                    ┌─────────────────────┐
                    │   分布式 SSD（3FS）   │
                    └───┬─────────────┬───┘
          读 KV-Cache   │             │   几乎无流量
          (SNIC 饱和🔴) │             │   (SNIC 空闲🟢)
                        ↓             ↓
              ┌──────────────┐  ┌──────────────┐
              │ Prefill 节点  │  │ Decode 节点   │
              │ GPU 在等数据… │  │ SNIC 在摸鱼…  │
              └──────────────┘  └──────────────┘
```

* **Prefill 侧：** 每次新一轮对话，需要从 SSD 加载海量历史 KV-Cache（上下文数万 token，命中率 >95%）。所有 I/O 压力集中在 Prefill 节点的存储网卡（SNIC）上，持续饱和。
* **Decode 侧：** 存储网卡几乎空闲——它只需从 Prefill 引擎通过 RDMA 接收数据，自身的 SSD 通道完全没被利用。
* **结果：** Prefill GPU 空转等数据，整个系统的吞吐被 Prefill 侧的 SNIC 带宽卡住。

**DualPath 的作用：** 打破"只有 Prefill 引擎才能读 SSD"的思维定式，让 Decode 引擎也参与数据加载，通过计算网络（RDMA）将数据转发给 Prefill 引擎，从而**池化（Pooling）所有引擎的存储带宽**——所谓池化，就是把原本各节点**各管各的** SNIC 带宽，汇聚成一个**统一调度的带宽池**，哪里有空闲就从哪里读，不再受限于单侧瓶颈。

## 这个问题真实存在吗？

**真实存在，且在 Agent 时代是最紧迫的系统瓶颈。**

![Figure 2: Agent 轨迹示例。模型与外部环境（终端、代码解释器）进行多轮交互，上下文逐轮累积。](https://arxiv.org/html/2602.21548v1/x2.png)

* **Agent 工作负载的独特性：** 与传统聊天不同，Agent 会与外部环境（终端、浏览器、代码解释器）进行**数十甚至数百轮**交互。每一轮只追加少量 token（工具输出或用户反馈），但上下文不断累积：

```
轮次1: [系统提示 + 用户指令]              → Agent 生成动作    上下文: 2K token
轮次2: [轮次1上下文 + 工具输出(~400 tok)] → Agent 生成动作    上下文: 3K token
轮次3: [轮次2上下文 + 工具输出(~400 tok)] → Agent 生成动作    上下文: 4K token
  ...                                      ...
轮次157: [累积上下文 + 工具输出(~400 tok)] → Agent 生成动作   上下文: 32K token
         ├─────── 98.7% 命中缓存 ────────┤├ 1.3% 新计算 ┤
```

DeepSeek 的生产数据显示，平均交互轮次高达 **157 轮**，每轮仅追加约 429 个 token，但上下文已积累到 3.2 万 token。这意味着 **98.7% 的 token 可以命中 KV-Cache**，只有极少量需要重新计算——推理系统的主要工作从"计算"变成了"搬数据"。
* **I/O-计算比极度失衡：** 对于 DeepSeek-V3.2 模型，在 Agent 场景下的 Cache-计算比高达 **22 GB/PFLOP**——每消耗 1 PFLOP 的算力，就需要从存储加载 22 GB 的 KV-Cache。对于 Qwen2.5-32B 这种 KV-Cache 更大的模型，这个比值更是飙到 117-267 GB/PFLOP。
* **硬件趋势在恶化：** 从 NVIDIA Ampere 到 Blackwell，GPU 算力增长了 14.4 倍，但网卡带宽的增长远远跟不上。**I/O-计算比在持续恶化**，这意味着问题只会越来越严重。

![Figure 3: 左图：NVIDIA GPU 硬件趋势（算力增速远超带宽增速）。右图：不同请求批量大小下的相对 token 吞吐量。](https://arxiv.org/html/2602.21548v1/x3.png)

## 为什么现在才有人去解决？

这个问题其实是伴随 **Agent 范式的爆发**和**推理架构的演进**才浮出水面的：

1. **Agent 范式的兴起：** 过去 LLM 主要用于单轮或少轮对话，上下文短，KV-Cache 复用率低，存储带宽根本不是瓶颈。随着 Coding Agent、自主任务 Agent 等应用兴起，多轮、长上下文、高缓存命中率成了新常态。
2. **PD 分离架构的普及：** PD 分离是近一两年才成为主流的推理范式。在此之前，Prefill 和 Decode 混在一起跑，没有明显的"一侧饱和、一侧空闲"问题。PD 分离提升了效率，但也**暴露了存储带宽分配不均**的结构性缺陷。
3. **外部 KV-Cache 存储的必要性：** Agent 的上下文动辄百万 token，服务器主内存（DRAM）缓存装不下（尤其在 RL 训练的 rollout 阶段，DRAM 被模型状态占满）。必须依赖更大更便宜的固态硬盘（SSD）存储（如 DeepSeek 的 3FS 分布式文件系统），但 SSD 读取带宽远低于 DRAM，问题由此凸显。

## 它是如何解决的？

DualPath 的核心思想可以概括为一句话：**"别让 Decode 引擎的存储网卡闲着，让它也帮忙从 SSD 搬数据。"**

### 1. 双路径加载（Dual-Path Loading）

这是整个系统的核心创新。对比传统方案和 DualPath：

```
【传统方案：单路径加载 —— Prefill SNIC 成瓶颈】

┌─────────┐      SNIC(饱和🔴)      ┌──────────────┐
│   SSD   │ ═══════════════════════→│ Prefill DRAM │──→ Prefill GPU
└─────────┘                         └──────────────┘
                                    ┌──────────────┐
                  SNIC(空闲🟢)      │ Decode DRAM  │    （完全没用上）
                                    └──────────────┘

总可用带宽 = Prefill 节点 SNIC 带宽（已打满）
```

```
【DualPath：双路径加载 —— 池化所有 SNIC 带宽】

              路径①：PE Read（传统路径）
┌─────────┐      SNIC              ┌──────────────┐
│         │ ───────────────────────→│ Prefill DRAM │──→ Prefill GPU
│   SSD   │                         └──────────────┘        ↑
│         │      SNIC              ┌──────────────┐    RDMA │
│         │ ───────────────────────→│ Decode DRAM  │─────────┘
└─────────┘                         └──────────────┘
              路径②：DE Read（新路径，借道 Decode 节点）

总可用带宽 = Prefill SNIC + Decode SNIC（翻倍！）
```

两条路径**并行工作**，由调度器动态决定每个请求走哪条路。路径②虽然多了一跳（Decode DRAM → RDMA → Prefill GPU），但**解锁了 Decode 侧闲置的 SNIC 带宽**，系统总吞吐大幅提升。

![Figure 4(a): PE Read Path — 传统路径，KV-Cache 从 SSD 直接加载到 Prefill 引擎。](https://arxiv.org/html/2602.21548v1/x4.png)

![Figure 4(b): DE Read Path — 新路径，KV-Cache 先加载到 Decode 引擎，再通过 RDMA 转发给 Prefill 引擎。调度器在两条路径间动态分配流量。](https://arxiv.org/html/2602.21548v1/x5.png)

> **RDMA（Remote Direct Memory Access，远程直接内存访问）** 是一种让机器 A 的网卡**直接读写**机器 B 内存的技术，全程绕过双方的 CPU 和操作系统。普通网络传输需要经过"应用 → 内核 → 网卡 → 网卡 → 内核 → 应用"的层层拷贝，而 RDMA 是"网卡 → 网卡"一步到位，延迟低至微秒级（约 1μs），带宽可达 400Gbps。AI 数据中心的 GPU 间通信（InfiniBand 计算网络）就是基于 RDMA，这也是 DualPath 敢让 Decode 节点"绕路"帮忙搬数据的底气——RDMA 这一跳快到几乎可以忽略。

### 2. 无瓶颈分析（Bottleneck-Free Analysis）

论文给出了严格的数学证明：在常见的 P/D 比例范围内，双路径加载**不会导致计算网卡或内存成为新瓶颈**。

以典型配置（每节点 8 个 GPU、1 个 SNIC）为例，无瓶颈的 P/D 比例范围是 $\frac{1}{7} \leq P/D \leq \frac{7}{2}$，覆盖了几乎所有实际部署场景。

### 3. CNIC 中心化流量管理

双路径意味着 KV-Cache 传输流量会和模型推理通信（如 AllToAll、AllGather）共享计算网络。如果不加隔离，搬运数据可能会干扰推理的延迟敏感操作。

DualPath 的解决方案是**把所有 GPU 进出数据都走 CNIC（计算网卡）**，利用 InfiniBand 的虚拟通道（Virtual Lane）实现硬件级流量隔离：

```
                  CNIC（计算网卡）
        ┌──────────────────────────────┐
        │                              │
        │  VL 0 [高优先级 ~99% 带宽]    │  ← 模型推理通信（AllToAll / AllGather）
        │  ██████████████████████████  │    亚毫秒级突发，绝对优先
        │                              │
        │  VL 2 [低优先级 ~1% 保底]     │  ← KV-Cache 搬运
        │  █                           │    见缝插针，利用推理间歇
        │                              │
        └──────────────────────────────┘

时间线上的效果：
 ───推理通信──┤  空闲  ├──推理通信──┤  空闲  ├──推理通信──
              ├KV搬运──┤            ├KV搬运──┤
              ↑                     ↑
         在间歇期搬运数据，完全不干扰推理
```

这确保了 KV-Cache 搬运**不会影响推理延迟**，同时充分利用了计算网络的空闲带宽。

### 4. 自适应请求调度（Adaptive Request Scheduler）

调度分为两级，分别解决"请求分给谁"和"GPU 一次算哪些"的问题：

```
                         请求队列
                     [R1, R2, R3, R4, ...]
                             │
               ┌─────────────┼─────────────┐
               │      第一级：引擎间调度     │
               │  决定分配到哪对 PE-DE 引擎  │
               │  以及走 PE Read / DE Read  │
               └─────┬───────────────┬─────┘
                     │               │
          ┌──────────▼────┐  ┌───────▼─────────┐
          │ PE-DE 引擎对 A │  │ PE-DE 引擎对 B   │
          │  R1, R3       │  │  R2, R4          │
          └──────┬────────┘  └──────┬──────────┘
                 │                  │
          ┌──────▼──────────────────▼──────┐
          │      第二级：引擎内调度          │
          │  决定本轮 Forward Batch 包含谁  │
          │  按"计算配额"控制每轮 Attention  │
          │  计算量，确保各 GPU 负载均衡     │
          └────────────────────────────────┘
```

* **引擎间调度：** 同时平衡**三个维度**——GPU 计算负载（token 数）、磁盘读取队列长度、SNIC 网络流量。优先分配到磁盘队列短的节点，防止 SNIC 空闲。
* **引擎内调度：** 用 FIFO 顺序打包请求进 batch，通过**计算配额（Compute Quota）**预估 Attention 层执行时间。若某请求超出配额，则对其做 chunked prefill（只算一部分），确保同一并行组内各 GPU 负载均衡，减少同步等待气泡。

## 还有更好的解决方案吗？

DualPath 针对的是 **PD 分离架构下 Agent 推理的存储 I/O 瓶颈**，属于系统层面的优化。同领域有其他路线：

* **Mooncake（分布式 DRAM 缓存池）：** 将 KV-Cache 缓存在集群所有机器的 DRAM（主内存）中，形成一个大规模分布式内存池，并通过**亲和性调度（Affinity-Aware Scheduling）**提升命中率——所谓亲和性调度，就是尽量把同一个 Agent 会话的后续请求调度到**上一轮缓存还驻留在 DRAM 中的那台机器**上处理，避免跨机器读取或重新从 SSD 加载。优点是 DRAM 的读取速度（~500 GB/s）远快于 SSD（~50 GB/s），延迟极低。缺点有二：其一，DRAM 容量有限且成本远高于 SSD（约 10 倍价差），在线服务的工作集动辄数百 GB 到 TB 级，全放 DRAM 成本过高；其二，在 RL 训练的 rollout 阶段，DRAM 被模型优化器状态和奖励模型等数据占满，根本腾不出空间给 KV-Cache。
* **HCache（KV-Cache 压缩）：** 减少需要加载的 KV-Cache 数据量。与 DualPath 正交，可以叠加使用。
* **Strata（分层缓存）：** 通过 GPU 辅助 I/O 和缓存感知调度来优化分层存储。同样与 DualPath 正交。

**DualPath 的优势在于"不改数据，改通路"：** 它不需要压缩或丢弃任何 KV-Cache 数据，也不依赖昂贵的 DRAM 扩容，而是通过重新设计数据流动路径来池化已有资源。这种思路具有很强的通用性和可组合性。

## 关键词解析

### 1. PD 分离架构（Prefill-Decode Disaggregation）

这是当前 LLM 推理服务的**事实标准架构**（前文已详细解释其原理和数据流）。补充几个值得注意的点：

* **常见误解：** PD 分离不是把模型拆成"编码器"和"解码器"——decoder-only 模型没有编码器。它拆分的是**推理的两个阶段**（Prefill 和 Decode），因为它们对硬件的需求完全相反。
* **业界趋势：** DistServe（2024）和 Splitwise（2024）是最早提出 PD 分离的系统论文，目前 SGLang、vLLM 等主流框架均已支持。
* **DualPath 的切入点：** PD 分离提升了效率，但也引入了新的数据流瓶颈——KV-Cache 必须在 SSD、Prefill 集群、Decode 集群之间频繁搬运，存储网卡（SNIC）成了最薄弱的环节。

### 2. 分层预填充（Layerwise Prefill）

这是 DualPath 依赖的另一项关键技术。

* **问题：** 长上下文 Prefill 时，如果所有层的 KV-Cache 必须全部驻留在 GPU 显存中，显存很快就会被撑满，限制了 batch size 和 GPU 利用率。

* **解决：** 由于 Prefill 计算有很强的局部性——每一层只需要当前层的 KV-Cache——可以**逐层加载、逐层释放**：

```
【传统 Prefill：所有层 KV-Cache 同时驻留显存】

  GPU HBM: [Layer1 KV][Layer2 KV][Layer3 KV]...[LayerN KV]  ← 显存被占满，batch size 受限

【分层 Prefill：同一时刻只保留一层的 KV-Cache】

  时刻1: 加载 Layer1 KV → 计算 Layer1 → 释放 Layer1 KV
  时刻2: 加载 Layer2 KV → 计算 Layer2 → 释放 Layer2 KV
  时刻3: 加载 Layer3 KV → 计算 Layer3 → 释放 Layer3 KV
  ...

  GPU HBM: [仅当前层 KV]  ← 显存占用降到 1/N，batch size 可扩大 N 倍
```

* **与 DualPath 的关系：** 分层预填充使得 KV-Cache 加载可以和计算**流水线化**——计算第 $k$ 层时，同时从 SSD 预加载第 $k+1$ 层的数据，为双路径加载创造了时间窗口。

### 3. 存储层级：HBM → DRAM → SSD（深入）

前文速查表已列出三者的定义，这里展开它们的**速度和成本差异**：

```
越靠近 GPU → 越快、越贵、越小 ▲
                                │
┌────────────┬────────────┬───────────┬───────┬──────────────────────────┐
│    层级     │    速度     │   容量     │ 成本  │        存什么             │
├────────────┼────────────┼───────────┼───────┼──────────────────────────┤
│  HBM       │  ~3 TB/s   │ 80-192 GB │ $$$$$ │ 模型权重 + 当前层 KV-Cache │
│  (GPU显存)  │            │           │       │                          │
├────────────┼────────────┼───────────┼───────┼──────────────────────────┤
│  DRAM      │  ~500 GB/s │ 数百GB~TB  │  $$$  │ 近期可复用的 KV-Cache 缓存 │
│  (主内存)   │            │           │       │                          │
├────────────┼────────────┼───────────┼───────┼──────────────────────────┤
│  SSD       │  ~7 GB/s   │ 数十TB+    │   $   │ Agent 海量历史 KV-Cache   │
│  (固态硬盘) │ ~50(3FS)   │           │       │                          │
└────────────┴────────────┴───────────┴───────┴──────────────────────────┘
                                │
越远离 GPU → 越慢、越便宜、越大 ▼
```

* **HBM（High Bandwidth Memory，高带宽显存）：** GPU 自带的显存，存放模型权重和当前正在计算的 KV-Cache。
* **DRAM（Dynamic Random-Access Memory，动态随机存取内存）：** 即服务器的主内存（"内存条"），可以用来缓存近期可能复用的 KV-Cache。
* **SSD（Solid State Drive，固态硬盘）：** 持久化存储，Agent 的海量历史 KV-Cache 最终存放在这里。

DualPath 要解决的就是**从最慢的 SSD 往最快的 HBM 搬数据时的带宽瓶颈**。

### 4. Cache-计算比（Cache-Compute Ratio）

这是论文用来量化"I/O 瓶颈有多严重"的核心指标。

* **定义：** 每消耗 1 PFLOP 计算量，需要从存储加载多少 GB 的 KV-Cache，单位是 **GB/PFLOP**。
* **直觉理解：** 这个值越高，说明 GPU 在算力上"越闲"，瓶颈完全在"搬数据"上。
* **实际数据：** DeepSeek-V3.2 在 Agent 场景下的值为 13-36 GB/PFLOP（MLA 优化后已经很小），Qwen2.5-32B 高达 117-267 GB/PFLOP。这说明对于 GQA 模型，存储带宽瓶颈尤其严重。

### 5. CNIC 与 SNIC（深入）

前文速查表已给出定义，这里展开它们在一个典型 8-GPU 节点中的**物理拓扑**：

```
┌─────────────────── 一个 GPU 节点 ───────────────────────┐
│                                                          │
│  GPU0  GPU1  GPU2  GPU3  GPU4  GPU5  GPU6  GPU7         │
│   │     │     │     │     │     │     │     │           │
│  CNIC  CNIC  CNIC  CNIC  CNIC  CNIC  CNIC  CNIC        │  ← 每 GPU 一张，400Gbps
│   │     │     │     │     │     │     │     │           │
│   └─────┴─────┴─────┴─────┼─────┴─────┴─────┘           │
│                           │                              │
│              ┌────────────┴────────────┐                 │
│              │  InfiniBand 计算网络     │  ← GPU 间通信（AllToAll 等）│
│              └─────────────────────────┘                 │
│                                                          │
│                        SNIC × 1                          │  ← 整个节点共享一张，400Gbps
│                          │                               │
│              ┌───────────┴────────────┐                  │
│              │  存储网络（连接 3FS）    │  ← 读写 KV-Cache / 模型检查点
│              └────────────────────────┘                  │
└──────────────────────────────────────────────────────────┘
```

* **CNIC（Compute NIC，计算网卡）：** 每个 GPU 配一张 400Gbps RDMA 网卡，走 InfiniBand 计算网络。总带宽 = 8 × 400Gbps = 3.2Tbps。
* **SNIC（Storage NIC，存储网卡）：** 每个**节点**只有一张 400Gbps 网卡，所有 8 个 GPU 共享，走独立的存储网络。
* **关键洞察：** 计算网络和存储网络**物理隔离**，CNIC 总带宽是 SNIC 的 **8 倍**。DualPath 的核心创新就是让 SNIC 的负载通过 CNIC 来均衡——用计算网络的空闲带宽弥补存储网络的不足。

### 6. 虚拟通道隔离（Virtual Lane Isolation）

这是保证 DualPath 不干扰推理性能的关键工程手段。

* **问题：** KV-Cache 传输和模型推理通信共享计算网络和 PCIe 通道。模型通信是亚毫秒级的突发流量，如果被 KV-Cache 传输"堵住"，延迟将不可接受。
* **解决：** 利用 InfiniBand 的 Virtual Lane 机制，将流量分成高低优先级。推理通信占 ~99% 带宽保障，KV-Cache 传输只在间隙中"见缝插针"。
* **类比：** 就像高速公路的应急车道——平时让普通车辆（KV-Cache）通行，但救护车（推理通信）来了必须立刻让路。

## 还有哪些值得深挖的洞察？

### 1. 从"计算瓶颈"到"I/O 瓶颈"的范式转移

这篇论文最深刻的贡献不是 DualPath 系统本身，而是它清晰地揭示了一个趋势：**Agent 时代的 LLM 推理，瓶颈已经从"算不过来"变成了"搬不动数据"。**

* **传统认知：** GPU 算力是稀缺资源，推理优化的核心是减少计算量（量化、剪枝、稀疏注意力等）。
* **新现实：** 在 Agent 场景下，98%+ 的 token 命中缓存，GPU 大部分时间在等数据。硬件趋势（算力增速 >> 带宽增速）让这个问题只会更严重。
* **启示：** 未来的推理系统设计需要从"以计算为中心"转向**"以数据流为中心"**。

### 2. "反直觉"的带宽池化：绕路反而更快

DualPath 的 DE Read Path 看起来很"绕"——数据从 SSD 先到 Decode 引擎，再通过计算网络转到 Prefill 引擎。常理上，绕路应该更慢。

* **为什么更快：** 因为瓶颈在 SNIC 总带宽，而非单次传输延迟。走 DE Read Path 虽然多了一跳，但**解锁了 Decode 侧闲置的 SNIC 带宽**。只要计算网络不成为新瓶颈（论文已证明），总吞吐量就会提升。
* **类比：** 就像城市交通——主干道堵死了，绕行辅路虽然远一点，但整体通行效率更高。关键不是单辆车的路程，而是整个路网的吞吐。

### 3. 硬件-软件协同设计的典范

DualPath 的成功高度依赖于现代 AI 数据中心的硬件特性：

* **计算网络与存储网络物理隔离：** 这使得"借道计算网络搬数据"在原理上可行。
* **InfiniBand VL / RoCE TC：** 硬件级 QoS 机制使得流量隔离可以在亚毫秒级别生效，软件层面根本做不到。
* **RDMA 的低提交延迟：** 论文指出 RDMA Write 的提交开销约 1μs，而 cudaMemcpyAsync 需要 5-7μs。这看似微小的差距，在海量小块传输场景下被放大了数倍。

这提醒我们：**顶层系统设计必须深入理解底层硬件拓扑**，而不是把网络和存储当作黑盒。

### 4. 与 Engram 的微妙联系

有趣的是，DualPath 和同为 DeepSeek 出品的 Engram 论文解决的是**同一枚硬币的两面**：

* **Engram** 解决的是**模型侧**的存储效率——用外部查找表替代 FFN 的死记硬背，减少模型参数对显存的占用。
* **DualPath** 解决的是**系统侧**的存储带宽——在推理时高效地搬运 KV-Cache，减少 GPU 的空转等待。

两者的哲学一脉相承：**把"存储"从"计算"中剥离出来，用更廉价的介质和更聪明的数据通路来承载。**

---

### 总结

DualPath 的本质是**"推理数据流的降本增效"**：

1. **双路径加载**打破了 Prefill 侧存储带宽的单点瓶颈。
2. **CNIC 中心化流量管理**确保搬数据不影响推理性能。
3. **自适应调度**实现了计算和网络资源的全局均衡。
4. **无瓶颈分析**从理论上保证了方案在常见配置下的可行性。

实验结果：离线推理吞吐提升最高 **1.87 倍**，在线服务吞吐平均提升 **1.96 倍**，并在 1152 GPU 规模下验证了近线性扩展能力。
