# Prompt 堆叠的尽头，是系统设计问题

在很多 AI 应用（尤其是 Code Review、PRD Review、Agent 类任务）中，一个非常常见的演进路径是：

> 通过不断增加 Prompt 规则，修复模型在某些 Case 上的失误。

短期来看，这种方式有效。但随着使用规模扩大，会逐渐进入一个稳定的困境：**Prompt 变得越来越长，但效果不再稳定提升，甚至开始下降。**

问题并不在于 Prompt 写得不够“细”，而在于它开始承担了不该由它承担的职责。

---

## 1. Prompt 的自然退化路径

一个典型的 Code Review Prompt 优化过程通常是这样的：

最初版本：

> 请深入分析代码中的潜在问题。

随后发现模型遗漏问题：

> 加入：不要遗漏潜在风险。

接着发现模型开始过度发挥：

> 加入：不要过度推测。

随后模型变得保守：

> 加入：适当分析潜在风险。

最终 Prompt 变成：

> 多分析，但不要猜；谨慎一点，但不要漏；深入一点，但不要过度。

这些约束并不是独立增加能力，而是在不断调节同一个空间里的张力。

本质上，它们在同时施加相互冲突的目标。

---

## 2. 问题的核心：冲突目标无法通过“补丁式 Prompt”解决

上述问题并不是 Prompt 不够精细，而是：

> 多个目标本身存在不可消解的冲突。

例如在 Code Review 中：

* 覆盖率 vs 精确率
* 深度分析 vs 避免臆测
* 严格检查 vs 避免噪音

这些目标之间不是“可以同时最大化”的关系，而是典型的 trade-off。

当 Prompt 用自然语言同时表达这些目标时，会出现三个问题：

### 2.1 规则不可执行化

“深入但不要过度推测”本质上没有可执行边界。

模型无法判断：

* 什么算“深入”
* 什么算“过度”

### 2.2 冲突不可调和

不同规则在不同上下文下会互相覆盖：

* 强调风险 → 误报增加
* 强调保守 → 漏报增加

Prompt 无法表达优先级动态变化。

### 2.3 系统行为不可预测

同一 Prompt 在不同代码片段中行为不一致，导致输出不可稳定评估。

---

## 3. 关键转折：从 Prompt Engineering 到 System Design

当 Prompt 同时承担以下职责时：

* 定义规则
* 定义边界
* 定义权衡
* 定义输出策略

问题就已经不再是 Prompt 问题，而是系统问题。

更具体地说，Prompt 在错误地承担三类职责：

### 3.1 Policy（策略层）

例如：

* 要不要深入分析
* 要不要保守
* 要不要优先性能

这些属于策略决策，而不是文本规则。

---

### 3.2 Evaluation（评估标准）

例如：

* 什么算高风险
* 什么算值得提示
* 什么算误报

这些需要 benchmark 或数据定义，而不是自然语言描述。

---

### 3.3 Knowledge（知识层）

例如：

* React useMemo 使用场景
* Go context 传递规则
* SQL injection 风险模式

这些应该进入 RAG 或结构化知识，而不是 Prompt 长文本。

---

## 4. Prompt 堆叠的本质问题

当不断通过“补丁式 Prompt”修复问题时，会出现一个结构性现象：

> Prompt 不再是规则集合，而变成“例外集合”。

典型演化路径：

* 规则 A
* A 的例外 A1
* A1 的例外 A1-1
* 与 B 冲突的情况
* 在 C 场景下失效的修正

最终 Prompt 变成：

> 一组不断相互修正的局部规则，而不是一致的系统行为描述。

这会直接导致三个后果：

* 难以维护
* 难以解释
* 难以评估

---

## 5. 一个更结构化的替代方式

要解决这个问题，关键不是“写更好的 Prompt”，而是拆分职责。

### 5.1 Prompt：定义角色与输出形式

Prompt 应该尽量稳定，例如：

> 扮演 Code Reviewer，对变更进行风险分析。

只保留职责边界，不承载策略细节。

---

### 5.2 RAG：提供按需知识

例如：

* 语言特定规则（Go / TS）
* 框架最佳实践
* 历史失败模式

按 diff 或上下文召回，而不是全量注入。

---

### 5.3 Evaluation：定义好坏标准

通过 benchmark 定义：

* false positive
* false negative
* review coverage
* noise ratio

用数据控制“深入 vs 保守”的平衡，而不是 Prompt 语言控制。

---

### 5.4 Observability：让行为可追踪

通过 tracing / logging：

* 哪些规则触发
* 哪些知识被召回
* 哪些判断产生

让系统行为变得可观测，而不是依赖直觉调 Prompt。

---

## 6. 结论

Prompt 堆叠的问题，本质不是“写得不够好”，而是：

> 当 Prompt 同时承载策略、知识与评估时，它必然走向冲突与膨胀。

所谓“不断加规则优化 AI”，在长期会退化成：

> 在一组无法调和的目标之间不断打补丁。

真正有效的路径是把系统拆开：

* Prompt 负责角色与边界
* RAG 负责知识
* Evaluation 负责权衡
* Observability 负责反馈

当这些职责分离之后，Prompt 才会重新变得稳定。

---

如果用一句话总结：

> **当 Prompt 开始描述“如何权衡多个冲突目标”时，问题已经不在 Prompt，而在系统设计本身。**
