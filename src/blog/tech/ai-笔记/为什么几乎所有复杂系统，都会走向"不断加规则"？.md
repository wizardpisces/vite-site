# 为什么几乎所有复杂系统，都会走向"不断加规则"？

前文 [《Prompt堆叠的尽头是系统设计》](https://mp.weixin.qq.com/s/AsTh9h-wnDeux0BrdLzKUQ)
讲了一个核心观点：**当 Prompt 开始堆叠描述“如何权衡多个冲突目标”时，问题已经不在 Prompt，而在系统设计本身。**

现在打算把这个观点往前再推一步；

规则的本质是堵住一个具体的漏洞。
反馈的本质是让系统自己知道发生了什么，然后自己调整。
规则解决“这次”，反馈解决“以后”。

## 不同领域的同一种病

**韩国教育的“钟摆困境”。‌** 

过去几十年，韩国对课外补习的政策来回摇摆了四五次。先禁止补习，地下补习泛滥；再限制时间，补习班改线上凌晨上课；又限制收费，机构拆分课程换个名目继续收。

每一次加规则，问题都换了个形式卷土重来。因为他们从没建立真正的反馈机制：学生到底需要什么？家长为什么焦虑？这些问题不回答，规则永远追不上变化。

**软件工程的 if-else 堆叠。‌** 

一个函数最初只有三行逻辑。后来 VIP 用户要特殊处理，加一个 if。新用户也要特殊处理，再加一个 if。某些地区要合规校验，再加一个 if。边界情况会崩溃，再加一个 if。一年后，这个函数变成了三十个 if-else 嵌套的怪物，没人敢改。

成熟的团队后来怎么做？不是继续加 if，而是重构——把策略抽成引擎，把规则抽成配置表，把校验抽成独立层。然后加一个关键的东西：日志。每一次判断都记录下来，出问题不需要猜，看一眼就知道哪一层出了问题。

**企业管理中的制度膨胀。‌** 

一家公司最初只有几条基本制度。后来有人报销超标，加一条审批规则。有人迟到，加一条考勤细则。有人泄露数据，加一条安全规范。三年后，员工手册从十页变成两百页，但违规行为并没有减少——只是变得更隐蔽。

真正有效的公司后来怎么做？不是加更多制度，而是建反馈通道：匿名上报、定期复盘、数据看板。让问题浮出水面，而不是用规则把问题压下去。

**AI 的 Prompt 堆叠**

当模型输出不理想时，最直接的调整方式仍然是规则层的修补。
增加 Prompt 约束。
补充行为说明。
添加例外条件。
这些手段通常可以改善局部表现。

但随着规则不断叠加，会逐渐出现另一类现象：
规则之间开始产生冲突。
目标之间开始互相制约。

## 为什么“加规则”会让人上瘾

因为增加规则是**成本最低**的改变方式。

```
不用调整组织。
不用修改架构。
不用重新设计流程。
只需要增加一句描述、一条制度、一个判断条件。
局部问题通常能够立刻得到改善。
于是规则越来越多。
```

而且它立竿见影。韩国教育部每次出台禁令，都能看到短期效果。程序员每次加一个 if，Bug 就修好了。管理者每次加一条制度，违规就收敛了。

但每一次“加规则”，都是在给系统增加一个“静态假设”——假设未来还会出现同样的问题，假设同样的规则还适用。

而现实是：问题不会重复，只会变形。你今天堵住一个漏洞，明天它会从另一个地方冒出来。你再加一条规则，它再变一次。最终，你得到的<u>不是一个更安全的系统，而是一个被规则捆死的系统。</u>

## 真正通用的准则

无论是写 Prompt、管团队、做产品、还是教育孩子，都适用：

不要问“我该加什么规则”，要问“我缺什么反馈”。‌

系统出错了，不是问“怎么禁止它再错”，而是问“为什么我不知道它错了”。团队效率低，不是问“怎么规定得更细”，而是问“为什么大家不知道优先级”。孩子成绩差，不是问“怎么管得更严”，而是问“他遇到什么困难没人知道”。

规则是“我知道答案，你照做”。反馈是“我不知道答案，但我知道怎么找到答案”。

## 更底层的视角

规则思维和反馈思维，背后是两种完全不同的世界观。

规则思维相信：**世界是确定的，问题可以穷举，只要规则足够多，就能覆盖所有情况。**

反馈思维相信：**世界是不确定的，问题是会演化的，唯一能依靠的是持续感知和快速调整的能力。**

前者追求“不出错”。后者追求“能纠错”。

而现实是：不出错是不可能的。能纠错才是真正的能力。

这也就是为什么，上一篇文章里讨论的 Prompt 堆叠问题，答案不是“写更好的 Prompt”，而是把策略、知识、评估这些职责拆出去，用 RAG、Evaluation、Observability 这些反馈机制来替代。Prompt 本身只保留角色定义，它不需要承载那么多。

这个逻辑，放在任何一个复杂系统里都成立。

## 最后

下次当你又想“加一条规则”的时候，停一下，问自己三个问题：

这个问题的根源，是我不知道，还是我没看到？
如果我不加规则，而是加一个反馈点，会怎样？
一年后回头看，这条规则还在吗？它真的有用吗？
